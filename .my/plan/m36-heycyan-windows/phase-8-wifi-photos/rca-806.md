# RCA-806: WiFi Transfer — Bypass BLE Command Discovery, Use Known Protocol

## Status: ACTIONABLE — New Approach

## Problem Statement

RCA-805 identified that the iOS SDK uses `openWifiWithMode:` to get WiFi credentials from the glasses. We attempted to reverse-engineer the exact BLE command bytes by testing hypotheses H1-H5. All tests returned **zero BLE notifications**, which we initially attributed to unknown command formats.

After deeper investigation, the zero-notification issue is a **BLE notification delivery problem on this Windows machine** (transient — previously worked fine). This is unrelated to the command format question.

## Key Insight: We Don't Need the Exact Command

The Android CyanBridge app successfully transfers photos using:
1. Send `{0x02, 0x01, 0x04}` (EnterTransferMode) over BLE
2. The glasses start a WiFi hotspot (or WiFi Direct group)
3. Connect to the hotspot
4. HTTP GET files from `http://<glasses-ip>:8080/`

The `openWifiWithMode:` command from iOS is a **convenience** — it returns the SSID and password so the app doesn't have to scan. But the underlying mechanism is the same: the glasses create a WiFi network after receiving the transfer mode command.

## Revised Approach: Scan-and-Connect

Instead of finding the exact `openWifiWithMode` bytes, we should:

1. **Send EnterTransferMode** `{0x02, 0x01, 0x04}` via BLE (already works)
2. **Scan WiFi networks** for the glasses hotspot (appears within ~10s)
3. **Connect with known password** `123456789` (confirmed from iOS source)
4. **Probe known IPs** for HTTP server: `192.168.43.1`, `192.168.4.1`, `192.168.1.1`
5. **Transfer files** via HTTP

## What We Know For Certain

### From Android CyanBridge (working, tested):
- Command `{0x02, 0x01, 0x04}` enters transfer mode
- Command `{0x02, 0x01, 0x09}` exits transfer mode
- After entering transfer mode, glasses expose a WiFi network
- HTTP server runs on port 8080
- Endpoints: `/filelist` (JSON listing), `/download?path=<file>`

### From iOS QCSDK (confirmed from headers + demo app):
- Default hotspot password: `123456789`
- Candidate IPs: `192.168.43.1`, `192.168.4.1`, `192.168.1.1`, `192.168.0.1`, `10.0.0.1`
- SSID pattern: likely contains device name or MAC suffix

### From Previous RCA-804 GATT Enumeration:
- Device name: `M01 Pro_E6C9`
- BLE MAC: `D8:79:B8:7F:E6:C9`
- MAC suffix: `E6C9`
- Possible SSID patterns: `M01*`, `*E6C9*`, `DIRECT-*`, `HeyCyan*`

### From Android WiFi Direct code (WifiP2pManagerSingleton.kt):
- Android uses WiFi Direct group owner with name containing `_E6C9`
- IP assigned to glasses as group owner: typically `192.168.49.1`
- But iOS uses standard hotspot mode — different mechanism, different IP range

## BLE Notification Issue (Separate Concern)

During testing, BLE notifications were not delivered despite:
- Connection status: `Connected`
- CCCD write: `Success`  
- Command writes: `Success`
- GattSession.MaintainConnection: active
- MaxPDU: 517

This is likely caused by:
- Glasses needing a reboot/reset after being in a bad state
- Or Windows BLE cache needing clearing after previous connection session
- Previously worked fine — not a fundamental protocol issue

**Resolution**: Reboot glasses (power off/on), clear Windows BT cache if needed, or pair via Windows Bluetooth Settings first. This does NOT block the WiFi scan approach.

## Implementation Plan

### Phase A: WiFi Hotspot Discovery (no BLE notifications needed)

```
1. Connect to glasses via BLE (GATT service de5bf728)
2. Write {0x02, 0x01, 0x04} to de5bf72a (enter transfer mode)
3. Wait 5-15 seconds for glasses to start hotspot
4. Use Windows.Devices.WiFi.WiFiAdapter.ScanAsync()
5. Find network matching: *M01*, *E6C9*, *DIRECT*, *HeyCyan*, *Glasses*
6. Connect with password "123456789" using WiFiAdapter.ConnectAsync()
7. Probe candidate IPs on port 8080 for HTTP response
8. Transfer files via /filelist and /download endpoints
9. Disconnect WiFi, reconnect to previous network
10. Write {0x02, 0x01, 0x09} to exit transfer mode
```

### Phase B: If Hotspot Not Found by Scan

- Try WiFi Direct enumeration (was previously broken but worth retrying)
- Try connecting to known SSID patterns manually
- Try `netsh wlan show networks` for hidden networks

### Phase C: Future — Proper openWifiWithMode (once BLE notifications work)

When BLE notifications are working again:
- Re-run RCA-805 Test 1 (extended capture) to verify notification delivery
- The response will contain SSID + password directly
- Eliminates need for WiFi scanning
