# RCA-808: EnsureClassicBtAudioAsync UnpairAsync Breaks BLE Notifications

## Status: FIX APPLIED — verification blocked by dead battery

## Problem Statement

BLE notifications from HeyCyan glasses stopped working after commit `c9e9890`
("ai glasses sound connection", May 17 15:20). All notification subscriptions
return CCCD Success, writes reach the device, but zero notifications are
delivered on any characteristic.

## Root Cause

Commit `c9e9890` introduced `EnsureClassicBtAudioAsync` which has an
**`UnpairAsync` nuclear fallback** that fires when the SDP query fails to
find audio endpoints within 30 seconds:

```csharp
// WindowsHeyCyanGlassesSession.cs — EnsureClassicBtAudioAsync Phase B
_log.LogWarning("EnsureBtAudio: profile forcing failed, trying unpair+repair");
var unpairResult = await classicDevice.DeviceInformation.Pairing.UnpairAsync()
    .AsTask(ct).ConfigureAwait(false);
```

For dual-mode devices like the M01 Pro glasses (BLE + Classic BT share the
same MAC address), calling `UnpairAsync()` on the Classic BT
`DeviceInformation.Pairing` **also invalidates the BLE bond**. Windows treats
dual-mode devices as a single logical pairing entity.

Once the BLE bond is broken:
- CCCD writes still return `Success` (BLE spec allows it for unbonded clients)
- But the GATT server firmware only enqueues notification PDUs for bonded peers
- Result: silent notification blackout

## Evidence

### Timeline correlation

| Time | Event | Notifications? |
|------|-------|---------------|
| May 17 02:15 | `5790ba2` committed (old `TryPairClassicAsync`, no unpair) | ✅ Working (81% battery) |
| May 17 15:20 | `c9e9890` committed (new `EnsureClassicBtAudioAsync` with UnpairAsync) | ❌ Broken |
| May 17–18 | All subsequent tests (RCA-804 through RCA-807) | ❌ Zero notifications |

### Code path that breaks notifications

```
ConnectAsync flow in c9e9890:
1. BLE GATT connect ✓
2. CCCD write (Notify) → Success ✓
3. ValueChanged += handler ✓
4. PairAsync (BLE) → BLE bond established ✓
5. EnsureClassicBtAudioAsync:
   a. Check existing endpoints → none (new feature, first run)
   b. Check Classic BT paired → yes (from step 4, dual-mode)
   c. TryForceProfileConnectionAsync → SDP query → poll 30s → FAILS
   d. ★ UnpairAsync() on Classic BT ← BREAKS BLE BOND ★
   e. TryPairClassicAsync → may timeout (no user interaction)
   f. Device left UNPAIRED
6. BLE notifications silently stop (bond gone)
```

### Why the old code worked

`5790ba2` only called `TryPairClassicAsync` which:
- Pairs if not paired
- Does nothing if already paired
- **Never unpairs**

## Fix

Remove the `UnpairAsync` nuclear option from `EnsureClassicBtAudioAsync`.
If profile forcing fails, log a warning and continue with GATT-only
connection. Never destroy an active BLE bond during a live session.

**Applied:** The `UnpairAsync` call has been removed from
`WindowsHeyCyanGlassesSession.cs`. The code now logs a warning and continues.

## Verification Status

**Cannot verify with hardware** — the glasses' battery is critically depleted.

### Diagnostic findings (battery-dead state)

The comprehensive BLE diagnostic (`_tmp/Program.cs`) proved:
- Device BLE radio is alive (connects, 9 GATT services visible)
- Static GATT reads work (firmware = AM01C_2.00.03_250718, hw = AM01C_V2.0)
- CCCD subscriptions accepted on all 8 notification characteristics
- Writes accepted on all write characteristics (de5bf72a, 6e400002, ae01)
- **ae01→ae02** (Jieli RCSP primary) delivers notifications — but ONLY echoes
- **NUS (6e400003), Serial Port (de5bf729)** — zero notifications regardless
  of command format (short HeyCyan, 16B OudMon framed, any content)
- Classic BT paired and RFCOMM services visible
- 60+ seconds passive wait: zero spontaneous notifications

### Interpretation

The Jieli chipset's BLE radio continues operating at minimum power (advertising,
GATT server, ae01 echo pass-through), but the **application MCU is shut down**.
The HeyCyan protocol (NUS + Serial Port services) runs on the application layer,
which requires sufficient battery to power the main processor.

This matches the timeline: battery was at 81% when notifications last worked,
then 39% at last successful read, and has been steadily draining through 50+
BLE connection cycles since.
