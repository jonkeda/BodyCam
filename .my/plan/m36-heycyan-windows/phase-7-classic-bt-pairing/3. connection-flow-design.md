# HeyCyan Windows Connection Flow — Full Design

**Date:** 2025-05-17  
**Status:** Proposed  
**Scope:** Bulletproof BLE + Classic BT connection for HeyCyan M01 glasses on Windows  
**Platform:** Windows-specific internals; shared layers remain cross-platform

---

## 1. Problem Statement

The current flow has a **fragile, linear dependency chain** that breaks when
any step's timing is off:

```
BLE scan → BLE connect → GATT setup → Classic BT pair → Audio endpoint appears → Router polls → Auto-select
```

Failure at any step after BLE connect leaves the user connected to the glasses
(GATT commands work, buttons work) but with **no audio**. The user sees:

```
Glasses mic unavailable — pair glasses as BT headset in Windows Settings for audio.
HeyCyan glasses mic registered but no BT capture endpoint found (mac=D8:79:B8:7F:E6:C9).
HeyCyan glasses speaker registered but no BT render endpoint found (mac=D8:79:B8:7F:E6:C9).
```

### Known Failure Modes

| # | Scenario | Current Behavior | Expected |
|---|----------|-----------------|----------|
| 1 | Classic BT already paired + connected | Works if endpoints exist at startup | Should work always |
| 2 | Classic BT already paired, NOT connected (audio) | Pairing succeeds immediately (`IsPaired=true`) but **no HFP/A2DP endpoints** appear — Windows doesn't auto-connect profiles | Should connect profiles |
| 3 | Classic BT not paired at all | `TryPairClassicAsync` runs 5 attempts; on success, A2DP usually appears, HFP may not | Should get both endpoints |
| 4 | Classic BT pair succeeds, endpoints slow to appear | Router polls 20s; `_mic.StartAsync` throws immediately | Should wait for endpoints |
| 5 | Intel SST routing (this PC) | Endpoints show `INTELAUDIO` not `BTHENUM`; requires name-cache fallback | Handled, but fragile |
| 6 | `MainPage.Loaded` not yet fired (deep-link to Settings) | Paired device cache empty → SST fallback fails | Should initialize eagerly |
| 7 | Glasses re-paired during app lifetime | Stale cache, stale `MMDevice` COM objects | Should recover |

**Scenario #2 is the current bug.** The glasses are paired from a previous
session, `IsPaired=true` so `TryPairClassicAsync` returns immediately, but
Windows hasn't connected the audio profiles. No HFP/A2DP endpoints exist in
`MMDevice` enumeration.

## 2. Cross-Platform Architecture Context

The HeyCyan connection stack is already layered for cross-platform use.
This design must **not** break that layering.

```
┌───────────────────────────────────────────────────────────────────┐
│                   Shared (cross-platform)                        │
│                                                                   │
│  GlassesViewModel ──> HeyCyanGlassesDeviceManager                │
│                         │                                         │
│                         ├─ _session : IHeyCyanGlassesSession      │
│                         ├─ _mic    : HeyCyanAudioInputProvider    │
│                         ├─ _camera : HeyCyanCameraProvider        │
│                         └─ _button : HeyCyanButtonProvider        │
│                                                                   │
│  HeyCyanAudioRouter (subscribes to session state, routes audio)  │
└────────────────────────────┬──────────────────────────────────────┘
                             │ IHeyCyanGlassesSession
        ┌────────────────────┼──────────────────────┐
        ▼                    ▼                      ▼
 ┌──────────────┐   ┌───────────────┐   ┌──────────────────┐
 │   Windows    │   │    Android    │   │       iOS        │
 │ WindowsHey-  │   │ AndroidHey-   │   │ IosHeyCyanGlasses│
 │ CyanGlasses  │   │ CyanGlasses   │   │ Session          │
 │ Session      │   │ Session       │   │                  │
 │              │   │               │   │ CoreBluetooth +  │
 │ BLE + Classic│   │ HeyCyan SDK   │   │ QCSDKManager     │
 │ BT pairing + │   │ bridge +      │   │ (no explicit     │
 │ SDP + MMDev  │   │ SCO audio     │   │  BT pairing)     │
 │ polling      │   │ routing       │   │                  │
 └──────────────┘   └───────────────┘   └──────────────────┘
```

### Key Constraint

`HeyCyanGlassesDeviceManager.ConnectAsync` is **shared code**. It calls
`_session.ConnectAsync(device, ct)` where `_session` is
`IHeyCyanGlassesSession` — the platform-specific implementation.

All Windows-specific BT audio logic (`EnsureClassicBtAudioAsync`, SDP
queries, MMDevice polling) must live **inside**
`WindowsHeyCyanGlassesSession.ConnectAsync`, not in the shared DeviceManager.

### Platform BT Audio Comparison

| Concern | Windows | Android | iOS |
|---------|---------|---------|-----|
| BLE connection | WinRT `BluetoothLEDevice` | HeyCyan SDK bridge | CoreBluetooth `CBCentralManager` |
| Audio pairing | Explicit Classic BT pair (the bug) | Implicit via SDK; SCO on-demand | Implicit via `AVAudioSession` |
| Audio endpoint discovery | MMDevice enumeration (WASAPI) | `AudioManager` bonded device scan | `AVAudioSession` route change |
| Endpoint timing issue | **Yes — this doc** | No (SCO connect is synchronous) | No (OS handles routing) |
| Profile force-connect needed | **Yes** (SDP query / unpair-repair) | No | No |

This design addresses the Windows column only. Android and iOS session
implementations already handle their BT audio correctly within their own
`ConnectAsync`.

### Race Condition: Two Parallel Paths

When `ConnectAsync` fires, TWO things happen concurrently:

1. **`DeviceManager.ConnectAsync`** calls `_mic.StartAsync(ct)` which calls
   `_bt.HasEndpointWithMac(mac)` — this checks if the BT enumerator already
   registered a provider for that MAC. If not → **throws immediately**.

2. **`HeyCyanAudioRouter.OnStateChanged(Connected)`** polls `IsAvailable`
   for up to 20 seconds — but it checks the HeyCyan provider's `IsAvailable`,
   which also delegates to `_bt.HasEndpointWithMac(mac)`.

The race: `_mic.StartAsync` fires first and throws before the Router even
starts polling. The catch in `DeviceManager` logs a warning and continues.
Then the Router polls for 20s and also fails.

Neither path triggers a **re-scan** of MMDevice endpoints after pairing.

## 4. Proposed Design: Bulletproof Connection Flow

### 4.1 Core Principle: Separate BLE and BT Audio Concerns

The BLE connection (GATT commands, buttons, camera) should **never** be
blocked or failed by BT audio issues. Audio is a **best-effort overlay** on
top of a successful BLE connection.

### 4.2 Where the Logic Lives

`EnsureClassicBtAudioAsync` is a **private method** inside
`WindowsHeyCyanGlassesSession`. It is called from the Windows session's
`ConnectAsync` — **not** from the shared `DeviceManager`.

```
WindowsHeyCyanGlassesSession.ConnectAsync(device, ct)
├─ BLE Phase (must succeed)
│   ├─ GATT service discovery
│   ├─ Characteristic subscription
│   └─ BLE bonding (optional)
│
├─ Classic BT Phase (best-effort, non-blocking)
│   ├─ EnsureClassicBtAudioAsync()  ← private Windows method
│   │   ├─ Step 1: Check if endpoints already exist → done
│   │   ├─ Step 2: If paired but no endpoints → trigger profile connect
│   │   ├─ Step 3: If not paired → pair, then trigger profile connect
│   │   └─ Step 4: Poll for endpoints with back-off
│   └─ Result: warning logged on failure; no return value to caller
│
└─ State = Connected
    (audio availability is discovered reactively by Router/providers)
```

The shared `HeyCyanGlassesDeviceManager.ConnectAsync` remains **unchanged**.
It calls `_session.ConnectAsync()` and then activates providers. If the mic
provider has no endpoint, it catches the exception — same as today, but now
the Windows session has done its best to make endpoints available first.

### 4.3 `EnsureClassicBtAudioAsync` — The New Unified Method

This replaces the current `TryPairClassicAsync` inside
`WindowsHeyCyanGlassesSession` with a method that handles all three
scenarios (already connected, paired but not connected, not paired).

```
// Private method inside WindowsHeyCyanGlassesSession
EnsureClassicBtAudioAsync(bleAddress, ct)
│
├─ Phase A: Quick Check — Are endpoints already here?
│   ├─ RefreshPairedDeviceCacheAsync()
│   ├─ ScanAndRegister() on both enumerators
│   ├─ HasEndpointWithMac(mac)?
│   └─ YES → return Success
│
├─ Phase B: Paired but no endpoints — Force profile connection
│   ├─ BluetoothDevice.FromBluetoothAddressAsync(bleAddress)
│   ├─ IsPaired?
│   │   ├─ YES → TryForceProfileConnectionAsync(device)
│   │   │        ├─ Disable/re-enable device to trigger Windows re-negotiation
│   │   │        ├─ OR toggle BT audio service via DeviceInformation
│   │   │        └─ Poll for endpoints (30s max)
│   │   │
│   │   └─ NO → go to Phase C
│   │
│   └─ Endpoints appeared? → return Success
│
├─ Phase C: Not paired — Full pairing sequence
│   ├─ TryPairClassicAsync (existing retry logic, 5 attempts)
│   ├─ On success → TryForceProfileConnectionAsync(device)
│   ├─ Poll for endpoints (30s max)
│   └─ return Success | PartialSuccess | Failure
│
└─ Phase D: Fallback notification
    ├─ If still no endpoints after all phases:
    │   ├─ Log warning with actionable guidance
    │   ├─ Raise event: ClassicBtAudioUnavailable
    │   └─ UI can show: "Open Windows Settings > Bluetooth to connect audio"
    └─ BLE connection remains active (commands, camera, buttons work)
```

### 4.4 Detailed Method: `TryForceProfileConnectionAsync`

When the device is paired but Windows didn't connect HFP/A2DP, we need to
kick Windows into connecting the profiles. Options in priority order:

#### Strategy 1: Device re-enumeration trigger (Preferred)

```csharp
private async Task<bool> TryForceProfileConnectionAsync(
    BluetoothDevice device, string mac, CancellationToken ct)
{
    // 1. Disconnect and reconnect the BT device to force profile negotiation.
    //    DeviceInformation.Pairing.UnpairAsync() + re-pair is nuclear —
    //    instead, we toggle the device's enabled state.
    var deviceInfo = device.DeviceInformation;
    
    // 2. Query RFCOMM services — this forces the SDP query which may
    //    trigger Windows to notice the HFP service and connect it.
    var hfpId = RfcommServiceId.FromUuid(
        Guid.Parse("0000111e-0000-1000-8000-00805f9b34fb"));
    var services = await device.GetRfcommServicesForIdAsync(hfpId)
        .AsTask(ct);
    
    if (services.Services.Count > 0)
    {
        _log.LogInformation("HFP service found via SDP — waiting for Windows to create endpoint");
        // The SDP query itself can trigger Windows BT stack to connect the profile.
        // Give it time.
    }
    
    // 3. Poll for endpoint appearance
    for (int i = 0; i < 15; i++) // 30 seconds max
    {
        await Task.Delay(2000, ct);
        await WindowsBluetoothEnumerator.RefreshPairedDeviceCacheAsync();
        _btEnumerator.ScanAndRegister();
        _btOutputEnumerator.ScanAndRegister();
        
        if (_btEnumerator.HasEndpointWithMac(mac) 
            || _btOutputEnumerator.HasEndpointWithMac(mac))
        {
            _log.LogInformation("BT audio endpoint appeared after {Seconds}s", (i + 1) * 2);
            return true;
        }
    }
    
    return false;
}
```

#### Strategy 2: Unpair + Re-pair (Nuclear Option)

If Strategy 1 fails, unpair and re-pair to force full profile negotiation:

```csharp
// Only use if Strategy 1 consistently fails
var unpairResult = await deviceInfo.Pairing.UnpairAsync().AsTask(ct);
if (unpairResult.Status == DeviceUnpairingResultStatus.Unpaired)
{
    await Task.Delay(2000, ct); // Let Windows clean up
    // Re-run Phase C pairing
    return await TryPairClassicAsync(bleAddress, ct);
}
```

#### Strategy 3: Open Windows Bluetooth Settings (User-Assisted Fallback)

```csharp
// Last resort — guide user to Windows Settings
await Launcher.LaunchUriAsync(new Uri("ms-settings:bluetooth"));
_log.LogWarning("Opened Windows Bluetooth Settings — user must manually connect audio profiles");
```

### 4.5 Updated `WindowsHeyCyanGlassesSession.ConnectAsync`

The Windows session's `ConnectAsync` now calls `EnsureClassicBtAudioAsync`
internally, after BLE setup and before setting state to Connected:

```csharp
// Inside WindowsHeyCyanGlassesSession (Windows platform only)
public async Task ConnectAsync(HeyCyanDeviceInfo device, CancellationToken ct)
{
    State = HeyCyanState.Connecting;
    Device = device;
    
    // Phase 1: BLE + GATT (must succeed)
    _bleDevice = await ConnectBleAsync(device, ct);
    await DiscoverGattServicesAsync(ct);
    await SubscribeCharacteristicsAsync(ct);
    
    // Phase 2: Classic BT audio (best-effort, Windows-only)
    // Replaces the old TryPairClassicAsync call.
    // EnsureClassicBtAudioAsync handles all scenarios:
    //   - Already connected → immediate return
    //   - Paired but no endpoints → SDP query + poll
    //   - Not paired → pair + SDP query + poll
    await EnsureClassicBtAudioAsync(device.Address, ct);
    // Failure is logged inside; does NOT throw.
    // Audio availability is discovered reactively by Router/providers.
    
    State = HeyCyanState.Connected;
    StateChanged?.Invoke(this, State);
}
```

### 4.6 `HeyCyanGlassesDeviceManager.ConnectAsync` — NO CHANGES

The shared `DeviceManager` remains cross-platform. It does **not** know
about `BtAudioResult` or `EnsureClassicBtAudioAsync`. The existing code
already handles mic unavailability gracefully:

```csharp
// Shared code — unchanged from current implementation
public async Task ConnectAsync(HeyCyanDeviceInfo device, CancellationToken ct)
{
    _lastDevice = device;
    await _session.ConnectAsync(device, ct);  // ← platform-specific

    Version = await _session.GetVersionAsync(ct);
    Battery = await _session.GetBatteryAsync(ct);
    await _session.SyncTimeAsync(ct);

    await _camera.StartAsync(ct);
    try
    {
        await _mic.StartAsync(ct);  // ← may throw if no BT endpoint
    }
    catch (InvalidOperationException ex) when (ex.Message.Contains("BT capture endpoint"))
    {
        _log.LogWarning("Glasses mic unavailable — ...");
    }
    await _button.StartAsync(ct);
    StatusChanged?.Invoke(this, EventArgs.Empty);
}
```

The improvement is that on Windows, by the time `_mic.StartAsync` runs, the
session has already done its best to ensure BT endpoints exist.

> **Persistence note:** After `_session.ConnectAsync` succeeds, the caller
> should save `device.Address` and `device.Name` for auto-reconnect on
> next startup. See [4. device-persistence-design.md](4.%20device-persistence-design.md).

### 4.7 Updated `HeyCyanAudioRouter` — Remove Duplicated Polling

The Router should **not** independently poll for endpoints. Instead, it should
react to endpoint availability events:

```
Current (broken):
  StateChanged(Connected) → poll IsAvailable for 20s

Proposed:
  StateChanged(Connected) → register providers, check IsAvailable once
  EndpointRegistered event → try auto-select (reactive, not polling)
```

The BT enumerators already have `IMMNotificationClient` handlers that fire
when devices are added/state-changed. Expose an event:

```csharp
// WindowsBluetoothEnumerator
public event Action<string>? EndpointRegistered;  // mac

// In ScanAndRegister, after registering a new provider:
EndpointRegistered?.Invoke(mac);
```

The Router subscribes:

```csharp
_btEnumerator.EndpointRegistered += OnBtEndpointRegistered;

private async void OnBtEndpointRegistered(string mac)
{
    if (_session.State != HeyCyanState.Connected) return;
    if (_session.Device?.Address != mac) return;
    
    // Auto-select if not already on glasses
    if (_input.ActiveProviderId != "heycyan-glasses" && _inputProvider.IsAvailable)
        await _input.SetActiveProviderAsync("heycyan-glasses");
    if (_output.ActiveProviderId != "heycyan-glasses" && _outputProvider.IsAvailable)
        await _output.SetActiveProviderAsync("heycyan-glasses");
}
```

## 5. Initialization: Eager Paired-Device Cache (Windows)

### Problem

The paired device cache is only built in `MainPage.Loaded`. If the user
deep-links or if the app reconnects automatically on startup, the cache
may be empty when the BT enumerators scan.

### Fix

Move cache initialization to DI registration or `App.OnStart`:

```csharp
// MauiProgram.cs — during service resolution
builder.Services.AddSingleton<WindowsBluetoothEnumerator>(sp =>
{
    var e = new WindowsBluetoothEnumerator(...);
    // Cache will be populated before first use
    return e;
});

// App.xaml.cs or MauiProgram.cs — after build
_ = WindowsBluetoothEnumerator.RefreshPairedDeviceCacheAsync();
```

This ensures the cache is warm before any connection attempt.

> **Persistence note:** This eager cache initialization also serves the
> auto-reconnect flow described in
> [4. device-persistence-design.md](4.%20device-persistence-design.md) —
> the cache must be warm before auto-reconnect attempts to call
> `EnsureClassicBtAudioAsync` with a saved address.

## 10. Error Recovery Matrix

| Scenario | Detection | Recovery Action | Timeout |
|----------|-----------|----------------|---------|
| Endpoints already exist | `HasEndpointWithMac` = true | None needed | 0s |
| Paired, endpoints appear after SDP query | SDP query triggers profile | Poll until found | 30s |
| Paired, endpoints never appear | Poll exhausted | Log warning, continue GATT-only | 30s |
| Not paired, pair succeeds | `TryPairClassicAsync` = true | SDP query + poll | 45s total |
| Not paired, pair fails (5 attempts) | `TryPairClassicAsync` = false | Log warning, continue GATT-only | 15s + 30s |
| Stale COM object | `InvalidCastException` on `_device.State` | try-catch → `IsAvailable = false` | 0s |
| Intel SST routing | `INTELAUDIO` in instance path | Name-cache fallback | 0s |
| `MMDevice` added after initial scan | `IMMNotificationClient.OnDeviceAdded` | Auto re-scan + register | Reactive |

## 11. Internal Result Type (Windows Only)

`BtAudioResult` is used **only** inside `WindowsHeyCyanGlassesSession` to
decide how much logging/retry to do. It is **not** exposed on the
`IHeyCyanGlassesSession` interface.

```csharp
// Internal to WindowsHeyCyanGlassesSession
private enum BtAudioResult
{
    /// <summary>Both capture and render endpoints found.</summary>
    Success,
    
    /// <summary>Only render (speaker) endpoint found. Mic unavailable.</summary>
    PartialSuccess,
    
    /// <summary>No BT audio endpoints found. GATT-only connection.</summary>
    Failure
}
```

## 14. Open Questions

1. **Does the SDP query (`GetRfcommServicesForIdAsync`) actually trigger
   Windows to connect HFP?** This needs empirical testing on this PC. If
   not, Strategy 2 (unpair/re-pair) may be needed.

2. **Should we add a "Connect Audio" button in the UI?** Currently audio
   connection is fully automatic. A manual fallback button could call
   `EnsureClassicBtAudioAsync` independently.

3. **BLE address ≠ Classic BT address?** On M01 Pro they appear to be the
   same (`D8:79:B8:7F:E6:C9`). On some dual-mode devices the Classic MAC
   differs by ±1. The current name-based fallback handles this, but we
   should verify.

4. **Windows 10 vs 11 behavior?** The Intel SST routing and profile
   connection behavior may differ. Test on Windows 11 when available.

5. **Device persistence & auto-reconnect** is scoped to a follow-up phase.
   Design is in [4. device-persistence-design.md](4.%20device-persistence-design.md).
   The connection flow is intentionally stateless — `EnsureClassicBtAudioAsync`
   takes an address parameter and doesn't care where it came from.

6. **Android/iOS parity:** This doc only fixes Windows. Android and iOS
   session implementations already handle their BT audio inside their own
   `ConnectAsync`. If similar issues surface on those platforms, the same
   pattern applies: fix it inside the platform session, not in the shared
   `DeviceManager`.
