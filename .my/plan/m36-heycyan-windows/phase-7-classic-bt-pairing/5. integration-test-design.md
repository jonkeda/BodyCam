# HeyCyan Connection Flow — Real Hardware Test Design

**Date:** 2025-05-17  
**Status:** Proposed  
**Scope:** Real-device tests for verifying the Windows BLE + Classic BT connection flow  
**Depends on:** [3. connection-flow-design.md](3.%20connection-flow-design.md)

---

## 1. Problem Statement

The connection flow (doc 3) introduces `EnsureClassicBtAudioAsync` with
multiple phases (quick-check, SDP force-connect, unpair/re-pair). These
interact with the Windows BT stack, MMDevice enumeration, and real BLE
hardware in ways that **cannot be faked**. The three failure scenarios from
doc 3 §1 — already connected, paired-but-not-connected, and fresh pair —
must be validated against real glasses.

The existing `HeyCyanRealFixture` in `BodyCam.RealTests` targets Android and
currently throws `NotImplementedException` on Windows. We need a Windows real
fixture and connection-specific test cases.

---

## 2. Approach: Real Hardware, No Fakes

Everything in the test uses the **real production code path** — the same
`WindowsHeyCyanGlassesSession`, `WindowsBluetoothEnumerator`,
`HeyCyanAudioRouter`, `HeyCyanGlassesDeviceManager`, and `HeyCyanAudioInputProvider`
/ `HeyCyanAudioOutputProvider` that the app uses. The only difference is
the test constructs these objects manually (no MAUI DI container).

### What's Under Test

```
┌───────────────────── All REAL ─────────────────────────────────────┐
│                                                                    │
│  HeyCyanGlassesDeviceManager                                       │
│    ├─ WindowsHeyCyanGlassesSession (BLE + Classic BT)              │
│    ├─ HeyCyanAudioInputProvider → WindowsBluetoothEnumerator       │
│    ├─ HeyCyanAudioOutputProvider → WindowsBluetoothOutputEnumerator│
│    ├─ HeyCyanCameraProvider                                        │
│    └─ HeyCyanButtonProvider                                        │
│                                                                    │
│  HeyCyanAudioRouter (subscribes to session + enumerator events)    │
│                                                                    │
│  AudioInputManager / AudioOutputManager (provider registry)        │
│                                                                    │
│  WindowsBluetoothEnumerator ── MMDevice API (WASAPI, COM)          │
│  WindowsBluetoothOutputEnumerator ── MMDevice API                  │
└────────────────────────────────────────────────────────────────────┘

Only stubbed:
  - ISettingsService (InMemorySettingsService — no MAUI Preferences dep)
  - IHeyCyanHttpClientFactory (not needed for connection tests)
  - IMediaStore / IMediaDurationProbe (not needed for connection tests)
```

### Why Real

- `EnsureClassicBtAudioAsync` calls WinRT `BluetoothDevice.FromBluetoothAddressAsync`,
  `GetRfcommServicesForIdAsync`, `DevicePairing.PairAsync` — these cannot be
  faked without losing the behavior we're testing.
- MMDevice endpoint timing is the core of the bug (doc 3 §1 scenario #2).
  Only real hardware can validate that endpoints appear after SDP queries.
- Intel SST routing (this PC) needs real `MMDevice` enumeration to test the
  name-cache fallback path.

- HeyCyan M01 or M01 Pro glasses, powered on and discoverable

## 4. Windows Real Fixture

### 4.1 `WindowsHeyCyanRealFixture`

Constructs the full production object graph without MAUI DI:

```csharp
public sealed class WindowsHeyCyanRealFixture : IAsyncDisposable
{
    // Core services
    public WindowsHeyCyanGlassesSession Session { get; }
    public HeyCyanGlassesDeviceManager DeviceManager { get; }
    public HeyCyanAudioRouter Router { get; }
    
    // Audio infrastructure
    public AudioInputManager AudioInput { get; }
    public AudioOutputManager AudioOutput { get; }
    public WindowsBluetoothEnumerator BtEnumerator { get; }
    public WindowsBluetoothOutputEnumerator BtOutputEnumerator { get; }
    public HeyCyanAudioInputProvider MicProvider { get; }
    public HeyCyanAudioOutputProvider SpeakerProvider { get; }
    
    // Lightweight stubs (no MAUI dependency)
    public InMemorySettingsService Settings { get; }
    
    private WindowsHeyCyanRealFixture(/* all params */) { /* assign */ }
    
    public static async Task<WindowsHeyCyanRealFixture> CreateAsync()
    {
        var loggerFactory = LoggerFactory.Create(b => b.AddDebug().SetMinimumLevel(LogLevel.Trace));
        var settings = new InMemorySettingsService();
        
        // Audio managers with platform mic/speaker
        var audioInput = new AudioInputManager(
            new[] { new PlatformMicProvider() },
            loggerFactory.CreateLogger<AudioInputManager>());
        var audioOutput = new AudioOutputManager(
            new[] { new WindowsSpeakerProvider() },
            loggerFactory.CreateLogger<AudioOutputManager>());
        
        // BT enumerators (real MMDevice)
        var btEnum = new WindowsBluetoothEnumerator(audioInput, settings);
        var btOutEnum = new WindowsBluetoothOutputEnumerator(audioOutput);
        
        // Warm paired-device cache
        await WindowsBluetoothEnumerator.RefreshPairedDeviceCacheAsync();
        btEnum.ScanAndRegister();
        btOutEnum.ScanAndRegister();
        
        // Session (real WinRT BLE)
        var session = new WindowsHeyCyanGlassesSession(
            loggerFactory.CreateLogger<WindowsHeyCyanGlassesSession>(),
            btEnum, btOutEnum);
        
        // Providers
        var mic = new HeyCyanAudioInputProvider(session, btEnum,
            loggerFactory.CreateLogger<HeyCyanAudioInputProvider>());
        var speaker = new HeyCyanAudioOutputProvider(session, btOutEnum,
            loggerFactory.CreateLogger<HeyCyanAudioOutputProvider>());
        var camera = new HeyCyanCameraProvider(session,
            new NullHeyCyanHttpClientFactory(),
            loggerFactory.CreateLogger<HeyCyanCameraProvider>());
        var button = new HeyCyanButtonProvider(session,
            loggerFactory.CreateLogger<HeyCyanButtonProvider>());
        
        // Router (real event-driven auto-select)
        var router = new HeyCyanAudioRouter(
            session, audioInput, audioOutput, mic, speaker,
            loggerFactory.CreateLogger<HeyCyanAudioRouter>());
        
        // Device manager (real orchestration)
        var deviceMgr = new HeyCyanGlassesDeviceManager(
            session, camera, mic, speaker, button, settings,
            loggerFactory.CreateLogger<HeyCyanGlassesDeviceManager>());
        
        return new WindowsHeyCyanRealFixture(
            session, deviceMgr, router,
            audioInput, audioOutput,
            btEnum, btOutEnum,
            mic, speaker, settings);
    }
    
    /// <summary>
    /// Scan for glasses and connect. Convenience for tests that don't
    /// care about the scan phase.
    /// </summary>
    public async Task<HeyCyanDeviceInfo> ScanAndConnectAsync(
        string mac, TimeSpan scanTimeout, CancellationToken ct)
    {
        var devices = await Session.ScanAsync(scanTimeout, ct);
        var target = devices.FirstOrDefault(d =>
            string.Equals(d.Address, mac, StringComparison.OrdinalIgnoreCase))
            ?? throw new InvalidOperationException(
                $"Glasses {mac} not found in scan. Found: {string.Join(", ", devices.Select(d => d.Address))}");
        
        await DeviceManager.ConnectAsync(target, ct);
        return target;
    }
    
    public async ValueTask DisposeAsync()
    {
        if (Session.State != HeyCyanState.Disconnected)
            await Session.DisconnectAsync(CancellationToken.None);
        await Session.DisposeAsync();
    }
}
```

### 4.2 `InMemorySettingsService`

Minimal in-memory implementation of `ISettingsService` to avoid MAUI
`Preferences` dependency in the test host. Only needs the settings properties
actually read/written during connection:

```csharp

    // Other ISettingsService members return defaults
    public string? ActiveAudioInputProvider { get; set; }
    public string? ActiveAudioOutputProvider { get; set; }
    // ... remaining properties with default values
}
```

## 6. Test Scenarios

### 6.1 Scan & Discovery

| Test | Precondition | Action | Assert |
|------|-------------|--------|--------|
| `Scan_FindsGlassesByMac` | Glasses on, discoverable | `Session.ScanAsync(10s)` | Returned list contains device with expected MAC |
| `Scan_DeviceNameMatchesExpected` | `BODYCAM_REAL_HEYCYAN_NAME` set | Scan | Device name starts with expected prefix |
| `Scan_RssiIsNegative` | Glasses nearby | Scan | `device.Rssi < 0` (valid BLE RSSI) |

### 6.2 Full Connection (doc 3 §13 test plan)

| Test | Precondition | Action | Assert | Timeout |
|------|-------------|--------|--------|---------|
| `Connect_FreshPair_FullAudio` | Unpair glasses in Windows Settings first | `ScanAndConnectAsync(mac)` | Both `MicProvider.IsAvailable` and `SpeakerProvider.IsAvailable` are true; `Session.State == Connected` | 60s |
| `Connect_AlreadyPaired_ImmediateAudio` | Glasses already paired + BT audio connected (prior session) | `ScanAndConnectAsync(mac)` | Endpoints available within 5s of connect call | 30s |
| `Connect_PairedNotConnected_AudioAppears` | Glasses paired but audio not connected (restart glasses, wait 10s) | `ScanAndConnectAsync(mac)` | Endpoints appear within 30s (SDP trigger) | 45s |
| `Connect_PopulatesVersionAndBattery` | — | Connect | `DeviceManager.Version != null`, `DeviceManager.Battery != null`, `Battery.Percentage > 0` | 30s |
| `Connect_SyncTime_DoesNotThrow` | — | Connect (includes `SyncTimeAsync`) | No exception | 30s |
| `Connect_SavesDeviceToSettings` | Empty `InMemorySettingsService` | Connect | `Settings.LastHeyCyanDeviceAddress == mac`, `Settings.LastHeyCyanDeviceName != null` | 30s |

### 6.3 Classic BT Audio Endpoint Validation

| Test | Precondition | Action | Assert |
|------|-------------|--------|--------|
| `BtEnumerator_FindsCaptureEndpointAfterConnect` | Connected | `BtEnumerator.HasEndpointWithMac(mac)` | `true` |
| `BtOutputEnumerator_FindsRenderEndpointAfterConnect` | Connected | `BtOutputEnumerator.HasEndpointWithMac(mac)` | `true` |
| `MicProvider_IsAvailableAfterConnect` | Connected | `MicProvider.IsAvailable` | `true` |
| `SpeakerProvider_IsAvailableAfterConnect` | Connected | `SpeakerProvider.IsAvailable` | `true` |
| `MicProvider_CanCaptureAudio` | Connected, mic available | `MicProvider.StartAsync()`, wait 2s | `AudioChunkAvailable` event fires at least once |

### 6.4 Audio Router Auto-Selection

| Test | Precondition | Action | Assert |
|------|-------------|--------|--------|
| `Router_AutoSelectsGlassesMicOnConnect` | — | Connect | `AudioInput.ActiveProviderId == "heycyan-glasses"` within 5s |
| `Router_AutoSelectsGlassesSpeakerOnConnect` | — | Connect | `AudioOutput.ActiveProviderId == "heycyan-glasses"` within 5s |
| `Router_RestoresPreviousOnDisconnect` | Connected, then disconnect | `Session.DisconnectAsync()` | `AudioInput.ActiveProviderId != "heycyan-glasses"` |

### 6.5 Disconnect & Reconnect

| Test | Precondition | Action | Assert |
|------|-------------|--------|--------|
| `Disconnect_StateIsDisconnected` | Connected | `Session.DisconnectAsync()` | `Session.State == Disconnected` |
| `Disconnect_MicBecomesUnavailable` | Connected, mic available | Disconnect | `MicProvider.IsAvailable == false` |
| `Reconnect_AfterDisconnect_Succeeds` | Connected → disconnected | Connect again | `Session.State == Connected`, endpoints available |
| `Reconnect_AudioEndpointsReappear` | Connected → disconnected → reconnected | Check endpoints | `BtEnumerator.HasEndpointWithMac(mac) == true` |

### 6.6 Edge Cases

| Test | Precondition | Action | Assert |
|------|-------------|--------|--------|
| `Connect_CancellationDuringBle_Throws` | — | Cancel token after 500ms | `OperationCanceledException` |
| `Connect_GlassesPoweredOff_TimesOut` | Glasses OFF, MAC still set | `ScanAsync(5s)` | Returns empty or no matching device |
| `IntelSst_EndpointsFoundViaNameCache` | Intel SST audio driver (this PC) | Connect | Endpoints found despite `INTELAUDIO` device path |

## 7. Test Structure Pattern

All tests follow the existing `BodyCam.RealTests` pattern:

```csharp
[Trait("Category", "RealConnection")]
public class WindowsConnectionFlowTests : IAsyncLifetime
{
    private readonly ITestOutputHelper _output;
    private WindowsHeyCyanRealFixture? _fixture;
    
    private static bool RealEnabled =>
        Environment.GetEnvironmentVariable("BODYCAM_REAL_HEYCYAN") == "1";
    private static string Mac =>
        Environment.GetEnvironmentVariable("BODYCAM_REAL_HEYCYAN_MAC") ?? "";
    
    public WindowsConnectionFlowTests(ITestOutputHelper output)
    {
        _output = output;
    }
    
    public async Task InitializeAsync()
    {
        if (!RealEnabled) return;
        _fixture = await WindowsHeyCyanRealFixture.CreateAsync();
    }
    
    public async Task DisposeAsync()
    {
        if (_fixture != null)
            await _fixture.DisposeAsync();
    }
    
    [SkippableFact]
    public async Task Connect_PopulatesVersionAndBattery()
    {
        Skip.IfNot(RealEnabled, "BODYCAM_REAL_HEYCYAN not set");
        Skip.If(string.IsNullOrEmpty(Mac), "BODYCAM_REAL_HEYCYAN_MAC not set");
        
        using var cts = new CancellationTokenSource(TimeSpan.FromSeconds(30));
        var device = await _fixture!.ScanAndConnectAsync(Mac, TimeSpan.FromSeconds(10), cts.Token);
        
        _output.WriteLine($"Connected to {device.Name} ({device.Address})");
        _output.WriteLine($"Version: {_fixture.DeviceManager.Version}");
        _output.WriteLine($"Battery: {_fixture.DeviceManager.Battery?.Percentage}%");
        
        _fixture.DeviceManager.Version.Should().NotBeNull();
        _fixture.DeviceManager.Battery.Should().NotBeNull();
        _fixture.DeviceManager.Battery!.Percentage.Should().BeInRange(1, 100);
    }
    
    [SkippableFact]
    public async Task Connect_BtAudioEndpointsAppear()
    {
        Skip.IfNot(RealEnabled, "BODYCAM_REAL_HEYCYAN not set");
        Skip.If(string.IsNullOrEmpty(Mac), "BODYCAM_REAL_HEYCYAN_MAC not set");
        
        using var cts = new CancellationTokenSource(TimeSpan.FromSeconds(60));
        await _fixture!.ScanAndConnectAsync(Mac, TimeSpan.FromSeconds(10), cts.Token);
        
        // EnsureClassicBtAudioAsync should have handled endpoint setup
        // before ConnectAsync returned. But poll briefly in case of async
        // endpoint registration via IMMNotificationClient.
        var endpointFound = false;
        for (int i = 0; i < 15 && !endpointFound; i++)
        {
            endpointFound = _fixture.BtEnumerator.HasEndpointWithMac(Mac);
            if (!endpointFound)
            {
                _output.WriteLine($"Waiting for BT capture endpoint... ({(i + 1) * 2}s)");
                await Task.Delay(2000, cts.Token);
            }
        }
        
        _output.WriteLine($"Capture endpoint: {_fixture.BtEnumerator.HasEndpointWithMac(Mac)}");
        _output.WriteLine($"Render endpoint: {_fixture.BtOutputEnumerator.HasEndpointWithMac(Mac)}");
        _output.WriteLine($"Mic available: {_fixture.MicProvider.IsAvailable}");
        _output.WriteLine($"Speaker available: {_fixture.SpeakerProvider.IsAvailable}");
        
        _fixture.BtEnumerator.HasEndpointWithMac(Mac).Should().BeTrue(
            "capture endpoint should exist after EnsureClassicBtAudioAsync");
        _fixture.MicProvider.IsAvailable.Should().BeTrue();
    }
}
```

## 8. Manual Precondition Tests

Some scenarios from doc 3 §13 require specific BT pairing states that can't
be set up programmatically. These are marked with `[SkippableFact]` and
require manual preparation:

| Test | Manual Step Before Running | Env Var Gate |
|------|---------------------------|-------------|
| `Connect_FreshPair_FullAudio` | Remove glasses from Windows Settings → Bluetooth → Paired Devices | `BODYCAM_REAL_HEYCYAN_FRESH_PAIR=1` |
| `Connect_PairedNotConnected_AudioAppears` | Restart glasses and wait 10s (paired but audio profiles not connected) | `BODYCAM_REAL_HEYCYAN_PAIRED_NO_AUDIO=1` |

These are intended for developer workstation runs, not CI.

## 10. Constructor Dependency Analysis

To construct the test fixture without MAUI DI, we need to verify each
service can be instantiated standalone:

| Class | Constructor Params | Test-Friendly? | Notes |
|-------|-------------------|----------------|-------|
| `WindowsHeyCyanGlassesSession` | `ILogger`, `WindowsBluetoothEnumerator`, `WindowsBluetoothOutputEnumerator` | Yes | All constructable |
| `WindowsBluetoothEnumerator` | `AudioInputManager`, `AppSettings` | Yes | `AppSettings` is a plain class |
| `WindowsBluetoothOutputEnumerator` | `AudioOutputManager` | Yes | Simple |
| `AudioInputManager` | `IEnumerable<IAudioInputProvider>`, `ILogger` | Yes | Pass empty or platform provider |
| `AudioOutputManager` | `IEnumerable<IAudioOutputProvider>`, `ILogger` | Yes | Same |
| `HeyCyanAudioInputProvider` | `IHeyCyanGlassesSession`, `IBluetoothAudioInputProvider`, `ILogger` | Yes | Needs `BluetoothAudioInputProvider` |
| `HeyCyanAudioOutputProvider` | `IHeyCyanGlassesSession`, `IBluetoothAudioOutputProvider`, `ILogger` | Yes | Needs `BluetoothAudioOutputProvider` |
| `HeyCyanAudioRouter` | `IHeyCyanGlassesSession`, `AudioInputManager`, `AudioOutputManager`, providers, `ILogger` | Yes | All above |
| `HeyCyanGlassesDeviceManager` | session, camera, mic, speaker, button, settings, `ILogger` | Yes | Camera/button can be stubs if needed |
| `HeyCyanCameraProvider` | session, `IHeyCyanHttpClientFactory`, `ILogger` | Needs stub factory | Camera not tested here |
| `HeyCyanButtonProvider` | session, `ILogger` | Yes | |

**Potential blockers:**
- `PlatformMicProvider` / `WindowsSpeakerProvider` — need to verify these
  can be constructed outside MAUI host (they use NAudio, not MAUI APIs)
- `BluetoothAudioInputProvider` / `BluetoothAudioOutputProvider` — need
  the `WindowsBluetoothEnumerator` instance; verify no circular deps

## 11. Diagnostic Output

Tests should log extensively via `ITestOutputHelper` to aid debugging when
run on different machines or after BT stack changes:

- BLE scan results (all devices found, not just target)
- Connection phase timing (BLE connect, GATT discovery, Classic BT pair, endpoint poll)
- MMDevice enumeration snapshot (all capture/render endpoints, not just BT)
- `EnsureClassicBtAudioAsync` result and timing
- Paired-device name cache contents
- Audio router state transitions

## 12. Open Questions

1. **Can `PlatformMicProvider` be constructed outside MAUI?** It uses NAudio
   `WaveInEvent` which should work, but may need a `SynchronizationContext`.
   If not, we can skip registering a platform provider and only test BT
   providers.

2. **Should the fixture reuse a single BLE connection across all tests in
   a class?** `IClassFixture<WindowsHeyCyanRealFixture>` would avoid repeated
   scan+connect overhead (~10-30s each). But some tests need specific BT
   states (disconnected, fresh pair). Recommend: `IAsyncLifetime` per test
   for isolation, with a shared-fixture variant for fast regression runs.

3. **How to handle flaky BT connections in CI?** These tests are
   developer-workstation only (`SkippableFact` + env var gate). They are NOT
   intended for CI. CI runs only `BodyCam.Tests` (unit) and
   `BodyCam.IntegrationTests` (WireMock).

4. **Windows 10 vs 11 differences?** The Intel SST routing test
   (`IntelSst_EndpointsFoundViaNameCache`) is specific to this dev machine.
   On Windows 11 with different audio drivers, the endpoint paths may differ.
   Tests should log enough to diagnose.
