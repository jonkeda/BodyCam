# HeyCyan Device Persistence & Auto-Reconnect — Design

**Date:** 2025-05-17  
**Status:** Proposed (future phase, after connection flow is stable)  
**Scope:** Remember last-connected HeyCyan device and auto-reconnect on app startup  
**Depends on:** [3. connection-flow-design.md](3.%20connection-flow-design.md) — `EnsureClassicBtAudioAsync` must be implemented first

---

## 1. Problem Statement

Today the user must **manually scan and reconnect** to their HeyCyan glasses
every time the app starts. The connection flow (doc 3) makes that connection
bulletproof, but doesn't address the UX friction of repeated manual pairing.

Additionally, if the app crashes while audio is routed to the glasses, the
"previous audio provider" snapshot is lost — the fallback provider on next
startup is undefined.

---

## 2. Current Persistence Inventory

| Component | Persisted? | Storage | Notes |
|-----------|-----------|---------|-------|
| Audio provider selection | Yes | `ISettingsService` (MAUI Preferences) | `ActiveAudioInputProvider`, `ActiveAudioOutputProvider` — provider IDs like `"bt:D8:79:B8:7F:E6:C9"` |
| Button mappings | Yes | `ISettingsService` (JSON in Preferences) | `ButtonMappingStore` — survives restarts |
| API keys | Yes | `SecureStorage` | Via `ApiKeyService` |
| HeyCyan device address/name | **No** | In-memory only (`_lastDevice`) | Lost on restart |
| HeyCyan device info (battery, firmware) | **No** | In-memory only | Lost on restart |
| Audio router fallback provider | **No** | In-memory only (`_previousInputId`) | Lost on crash |
| BT paired-device name cache | **No** | Static `ConcurrentDictionary` | Rebuilt at runtime |

---

## 3. Proposed Changes to `ISettingsService`

Add three new keys:

```csharp
// ISettingsService.cs
string? LastHeyCyanDeviceAddress { get; set; }  // BLE address, e.g. "D8:79:B8:7F:E6:C9"
string? LastHeyCyanDeviceName { get; set; }      // Display name, e.g. "M01 Pro"
bool HeyCyanAutoReconnect { get; set; }          // User toggle, default true
```

Implementation in `SettingsService.cs` follows the existing `Preferences.Get`/`Set` pattern.

---

## 4. When to Save

Save in `HeyCyanGlassesDeviceManager.ConnectAsync`, **after** BLE connects
successfully (not after BT audio — that's best-effort):

```csharp
// After _session.ConnectAsync succeeds
_settings.LastHeyCyanDeviceAddress = device.Address;
_settings.LastHeyCyanDeviceName = device.Name;
```

Clear on explicit user-initiated forget/unpair (if we add that), but
**not** on normal disconnect — the user expects to reconnect to the same
device next time.

## 5. When to Restore (Auto-Reconnect)

### 5.1 Startup Flow

```
App.OnStart / MainPage.OnAppearing
│
├─ Is HeyCyanAutoReconnect enabled?
│   └─ NO → stop, show normal scan UI
│
├─ Is LastHeyCyanDeviceAddress set?
│   └─ NO → stop, show normal scan UI
│
├─ Warm BT paired-device cache
│   └─ RefreshPairedDeviceCacheAsync()
│
├─ Attempt quick BLE scan for saved address (5s timeout)
│   ├─ Found → ConnectAsync(savedDevice, ct)
│   │          (EnsureClassicBtAudioAsync handles BT audio)
│   │
│   └─ Not found → show scan UI with toast:
│                   "Could not find {LastHeyCyanDeviceName}"
│
└─ On ConnectAsync failure → show scan UI, don't clear saved device
```

### 5.2 Key Design Decisions

- **Short scan timeout (5s):** Don't block the user for 30s if glasses are off.
  The normal scan UI is always reachable.
- **Don't clear on failure:** The saved device is still the user's device —
  they may have just forgotten to turn it on.
- **Skip scan UI on success:** Navigate directly from glasses page to main
  page, or show a brief "Reconnecting to M01 Pro..." overlay.

## 6. Audio Router Fallback Persistence

### Problem

`HeyCyanAudioRouter` snapshots the current audio provider before routing to
glasses (`_previousInputId`, `_previousOutputId`). These are in-memory only.
If the app crashes while routed to glasses, the fallback is unknown on restart.

### Non-Issue with Current Design

The `AudioInputManager` already persists `ActiveAudioInputProvider`. When
glasses disconnect, the router calls `SetActiveProviderAsync` with the
previous provider. On crash + restart, `AudioInputManager.InitializeAsync()`
restores whatever was last persisted — which would be `"heycyan-glasses"`.

Since the glasses aren't connected on restart, that provider won't be found,
and it falls back to `"platform"`. This is acceptable behavior — the user
gets their default mic back.

**Conclusion:** No additional persistence needed for the audio router fallback.
The existing `ActiveAudioInputProvider` persistence handles the crash case
well enough.

---

## 7. What NOT to Persist

| Data | Why Not |
|------|---------|
| Battery level | Stale instantly — re-read on connect |
| Firmware/hardware version | Changes rarely, re-read on connect |
| Media counts | Changes constantly, re-read on connect |
| BLE GATT handles | OS-specific, not portable, not stable |
| Classic BT pairing state | Windows manages this — just check `IsPaired` |

## 8. Interaction with Connection Flow (Doc 3)

The connection flow design is intentionally **stateless** —
`EnsureClassicBtAudioAsync(bleAddress, ct)` is a private method inside
`WindowsHeyCyanGlassesSession` that takes an address and handles BT audio
best-effort. It doesn't know or care whether that address came from:

- A fresh BLE scan (manual connect)
- A persisted `LastHeyCyanDeviceAddress` (auto-reconnect)

The shared `HeyCyanGlassesDeviceManager.ConnectAsync` is **not modified**
by the connection flow design — it remains cross-platform. Persistence
is the caller's responsibility (`GlassesViewModel` or a future
`AutoReconnectService`).

### Cross-Platform Note

This persistence design is **shared code** — it applies equally on Windows,
Android, and iOS. The settings (`LastHeyCyanDeviceAddress`, etc.) are stored
via `ISettingsService` which uses MAUI `Preferences` on all platforms.
Auto-reconnect calls the same `_session.ConnectAsync()` as manual connect;
each platform's session implementation handles its own BT audio internally.

### Touch Points in Doc 3

| Section | Impact |
|---------|--------|
| §5 Eager Paired-Device Cache | Auto-reconnect on Windows depends on cache being warm at startup. The `RefreshPairedDeviceCacheAsync()` call in `App.OnStart` serves both this doc and doc 3. |
| §4.6 DeviceManager.ConnectAsync (unchanged) | Persistence is added in the caller (`GlassesViewModel`) — save `device.Address` / `device.Name` after `ConnectAsync` succeeds. |

## 10. Testing Plan

| Test Case | Expected |
|-----------|----------|
| Connect → restart app → glasses on | Auto-reconnect within 5s, full audio |
| Connect → restart app → glasses off | 5s scan timeout, show scan UI, saved device intact |
| Connect → crash → restart | Falls back to platform mic, saved device intact |
| Toggle auto-reconnect off → restart | Shows scan UI, no auto-connect attempt |
| Forget device → restart | Shows scan UI, no saved device |
| Connect device A → connect device B → restart | Auto-reconnects to device B (last connected) |
