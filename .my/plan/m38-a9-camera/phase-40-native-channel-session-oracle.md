# Phase 40 - Native Channel Session Oracle

**Status:** Completed first pass - native transport still remains

## Roadmap Gate

This phase implements gate 2 from
[C#-Only Vue990 Stream Roadmap](./csharp-only-vue990-roadmap.md): use the
vendor native libraries as a measurement oracle so the C# implementation stops
guessing the session opener.

The output of this phase is not the final product path. The final path still
must be C# protocol code. Native calls are allowed here only to reveal packet
order, channel messages, and media payload shapes.

## Why This Phase Exists

Recent C# work improved the packet builders, but repeated broad probing is no
longer useful:

- Direct HTTP media paths keep returning no media.
- RTSP has no open port evidence.
- Classic PPPP/HLP2P discovery only finds self-echoes.
- Managed C# relay packets match known native vectors but live relays stay
  silent.
- A working native-backed Android stream does not expose a clear media socket
  through the current socket snapshots.

The next useful experiment is to observe the successful native session from
inside the same Android probe that already captures images and video.

## Goal

Record enough native-backed session evidence to implement the next managed C#
opener without vendor libraries.

## Plan

1. Inventory candidate native hooks.
   - Reuse the existing native packet oracle infrastructure.
   - List exported `HLP2P_*`, `PPCS_*`, `VEEPAI_*`, `JNIApi_*`, and packet
     helper symbols that can expose session bytes or state.
   - Prefer functions already called safely by `Vue990PpcsSession`.

2. Add bounded channel oracle logging.
   - During a working native-backed session, log `checkMode` and `checkBuffer`
     at fixed milestones.
   - Look for a safe native read/peek function that can copy channel bytes into
     a C# buffer without draining the player unexpectedly.
   - If only destructive reads are available, run this in a separate oracle
     mode and save raw bytes before playback.

3. Add native packet/session oracle calls.
   - Call native helpers on loopback or in a contained mode first.
   - Capture generated packet bytes, return codes, and any state-dependent
     values after `connect`, `login`, and live-open.
   - Compare generated bytes with the managed builders already implemented.

4. Write the managed target contract.
   - Document the exact packet/message sequence that C# must reproduce.
   - Identify which fields are stable, which are random/session-derived, and
     which require a response from the camera.

5. Stop at the first useful packet boundary.
   - This phase does not need to decode full video.
   - The phase is successful once it explains why the managed C# opener does
     not receive the first non-self camera packet or channel bytes.

## 2026-05-29 Result

Implemented a native channel oracle in the Android C# probe:

- Added a bounded P/Invoke hook for native `client_read`.
- Ran the normal native-backed session through:
  `JNIApi.connect -> JNIApi.login -> JNIApi.writeCgi`.
- Skipped `AppPlayerApi` so the player would not consume the channel bytes.
- Read channel `1` directly into C# buffers.
- Pulled the raw channel dumps back to Windows.
- Added a shared C# `A9Vue990ChannelMediaExtractor`.
- Extracted JPEG frames and assembled a MJPEG AVI on Windows with C#.

Artifacts:

- First raw oracle run:
  `.my/plan/m38-a9-camera/captures/phase-40-native-channel-oracle-2026-05-29-165948/`
- Media extraction run:
  `.my/plan/m38-a9-camera/captures/phase-40-channel-media-extract-2026-05-29-170337/`
- Still image:
  `.my/plan/m38-a9-camera/captures/phase-40-channel-media-extract-2026-05-29-170337/native-channel-oracle-frames/channel-frame-000.jpg`
- Video:
  `.my/plan/m38-a9-camera/captures/phase-40-channel-media-extract-2026-05-29-170337/native-channel-oracle-mjpeg.avi`

Observed:

- `client_read` returned raw channel bytes after the live CGI command.
- Raw chunks commonly start with:
  `55 AA 15 A8 03 03 00 00 ...`
- The channel envelope is `0x20` bytes followed by a normal JPEG payload.
- Some reads begin directly at `FF D8` when the read starts inside an already
  queued JPEG payload.
- Extracted `54` JPEG frames.
- First extracted frame is `640x480`, `8371` bytes, SHA-256
  `4E83B4815C061D0F6D6E3A10B381693869F716BFD890C6CE9AA836A0F97655F1`.
- C# assembled `native-channel-oracle-mjpeg.avi`, `452860` bytes, SHA-256
  `5EF54A34006270658E48B8D58279C5F5C57C6AACC80849CD8D9F6496E5E11827`.

Interpretation:

- The media payload is not H.264 for this live path; it is JPEG frames inside a
  small Vue990 channel envelope.
- `AppPlayerApi` is no longer required to save image/video once channel bytes
  are available.
- This is still not the final pure C# transport path because `JNIApi.connect`,
  `JNIApi.login`, `JNIApi.writeCgi`, and native `client_read` still provide the
  session transport.
- The next pure-C# step is to replace native pieces in order: first prove the
  C# live-CGI frame through native `JNIApi.write`, then replace native channel
  transport with the managed session opener.
