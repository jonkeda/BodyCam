# Phase 48 - Control Derivation And Windows Port

**Status:** Windows C# capture succeeded; final hardening continued in Phase 49

## Goal

Turn the Phase 47 Android C# runtime success into a maintainable cross-platform
C# capture path:

- derive or generate the encrypted post-hole `0D` control payloads in C#;
- expose the working direct capture sequence through shared code;
- run the same sequence from Windows when the laptop can reach the camera
  network or another equivalent network path is available.

## Starting Point

Phase 47 proved the camera stream can be captured by managed C# at runtime on
Android:

- compact LAN-hole response and ready are parsed in C#;
- direct `0D` command/media packets are parsed in C#;
- command/media ACKs are generated in C#;
- `55 AA 15 A8` media bytes are reassembled in C#;
- still image and MJPEG AVI are saved without native session calls.

Successful artifact directory:

```text
.my/plan/m38-a9-camera/captures/phase-47-managed-hlp2p-direct-paced-2026-05-30-001842/
```

The remaining weakness is that the four post-hole control payloads are fixed
encrypted vectors observed from the native app. They are sent from C#, but they
are not yet generated by C#.

## Work Plan

1. Stabilize the Android-proven path.
   - Move direct transport code toward shared library boundaries.
   - Keep the Android phone probe as a runnable proof harness.
   - Preserve the exact Phase 47 packet ordering as a regression path.

2. Derive post-hole control generation.
   - Compare native socket-hook runs that differ in LAN-hole nonce/token/status.
   - Separate stable plaintext packet fields from encrypted payload bytes.
   - Locate or reproduce the native cipher/key schedule used for `0D` control
     payloads.
   - Add golden tests for generated control packets.

3. Make capture API shape explicit.
   - Provide a C# call that returns still path, video path, frame count,
     dimensions, byte counts, and hashes.
   - Keep raw channel and packet dumps opt-in diagnostic outputs.

4. Port to Windows.
   - Use the same compact LAN-hole/direct `0D` sequence.
   - Only debug Windows Firewall/socket binding if Windows sends the
     known-good bytes and fails at a specific packet stage.
   - Save the same still and MJPEG AVI artifacts from Windows C#.

## Acceptance Criteria

- C# generates the post-hole controls or documents exactly why static encrypted
  vectors are acceptable for this camera class.
- Android C# capture still passes after moving code to shared boundaries.
- Windows C# can run the same sequence when Windows has a path to
  `192.168.168.1`.
- Windows C# saves a still image and MJPEG AVI.
