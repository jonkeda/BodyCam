# Phase 49 - Final C# Hardening

**Status:** Current-camera hardening succeeded; broader control derivation remains future work

## Goal

Turn the proven Android and Windows C# capture paths into a clean, repeatable,
maintainable implementation.

The practical stream goal is already achieved for `@MC-0025644`: C# can save a
still image and MJPEG AVI from Android and from Windows. This phase is the last
hardening step before treating the Vue990 path as a real library feature rather
than a successful reverse-engineering harness.

The direct runtime path now does these in C#:

- reaches camera HTTP status on `192.168.168.1:81`;
- performs the compact LAN-hole and ready flow;
- sends direct `0D` command/media ACKs;
- receives `55 AA 15 A8` media bytes;
- extracts JPEG frames;
- writes a still image and MJPEG AVI.

Remaining caveat:

- The post-hole control payloads are still static encrypted byte vectors
  observed from the native Vue990 socket trace. They are sent by C#, but not
  generated by C# yet.

## Work Plan

1. Freeze the known-good capture contract.
   - Keep Phase 47 and Phase 48 artifact paths as regression evidence.
   - Preserve the exact native-paced control order as a baseline.
   - Add a short developer-facing command example for Windows direct capture.

2. Inventory the static encrypted controls.
   - Name each post-hole control by role, timing, packet length, and observed
     camera response.
   - Record the packet bytes and the handshake state available before each
     control is sent.
   - Remove ambiguous "magic" wording from code comments and reports.

3. Derive or constrain the control generation.
   - Compare at least two successful direct capture runs with different
     LAN-hole tokens, ports, or session values if available.
   - Check whether the encrypted controls are stable per camera, per firmware,
     per user/password, or per session.
   - Inspect the native code path for the `0D` post-hole control payload
     cipher/key schedule if packet comparison is not enough.
   - Implement a C# control builder if derivation is found.
   - If derivation is not found, explicitly document the supported static-vector
     scope and failure modes.

4. Clean the public C# capture API.
   - Keep the shared direct capture implementation outside Android-only code.
   - Expose an options object for host, credentials, duration, frame limit,
     output directory, and diagnostic dump level.
   - Return a result object with success, error, still path, video path, frame
     count, dimensions, byte counts, hashes, and raw dump paths.
   - Keep packet dumps opt-in for normal use.

5. Prove repeatability.
   - Run Windows direct capture at least twice after cleanup.
   - Run Android direct capture once if the phone is available and still useful
     as a cross-platform check.
   - Save still and video artifacts for the final proof run.
   - Update the realtests log, roadmap, and current status report with final
     outcomes.

## Acceptance Criteria

- Windows C# direct capture still saves a JPEG and MJPEG AVI after cleanup.
- Android C# direct capture still works or is explicitly marked as a legacy
  proof harness if Windows becomes the only maintained runtime.
- The four post-hole controls are either generated in C# or documented as a
  deliberately scoped compatibility vector for this camera/firmware class.
- The capture API no longer exposes reverse-engineering names such as "replay"
  in normal call paths.
- Tests cover direct packet parsing, ACK generation, frame extraction, AVI
  writing, and any generated control payload logic.
- The final report contains exact artifact paths, hashes, dimensions, and the
  remaining risk, if any.

## Non-Goals

- Do not restart broad HTTP/RTSP/MJPEG probing.
- Do not reopen TCP relay work unless direct capture breaks and packet evidence
  points there.
- Do not add H.264/FFmpeg work while the camera is producing JPEG frames.
- Do not require the vendor Vue990 app for the final Windows capture path.
