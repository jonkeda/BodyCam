# 5.1 вҖ” Wake Word & Listening Modes вңҰ Deep Analysis

## Why the "Always-On Realtime API" Approach Is Wrong

The original DESIGN.md dismissed wake words because `semantic_vad` "already solves this."
That's technically true for *intent detection*, but ignores two brutal realities:

### Cost

GPT-realtime-1.5 pricing:
- **Audio input: $32.00 / 1M tokens**
- Audio output: $64.00 / 1M tokens
- Cached input: $0.40 / 1M tokens

Audio tokens accumulate at roughly **~50 tokens/second** of speech. An always-on
WebSocket streaming silence/ambient noise still sends keep-alive frames and the
session incurs costs for any audio the VAD picks up.

**Rough cost estimate for always-on:**
| Scenario | Daily audio in | Daily cost (input only) |
|----------|---------------|------------------------|
| 1 hour active use | ~180K tokens | ~$5.76 |
| 8 hours background | ~1.4M tokens | ~$44.80 |
| Always-on 16h/day | ~2.9M tokens | ~$92.80 |

Even if `semantic_vad` discards most audio server-side, **the WebSocket connection
itself keeps the session alive and billable**. The connection isn't free.

### Battery

A continuous WebSocket connection + mic streaming drains battery significantly:
- Microphone capture: ~50-100mW continuous
- WebSocket + network: ~100-200mW continuous
- Audio encoding/buffering: ~20-50mW
- **Total: ~170-350mW continuous drain**

Compare to wake word engine (Porcupine):
- On-device inference: ~5-10mW
- No network until triggered
- **10-30x less power draw than streaming**

Smart glasses have tiny batteries (typically 300-500mAh). Always-on streaming
would drain them in 2-3 hours. Wake word detection can run all day.

## Wake Word Strategy

### Primary Wake Word: "Hey BodyCam"
- Activates full conversational session (Layer 3)
- Opens WebSocket, starts streaming audio to Realtime API
- User can then talk naturally for as long as needed
- Session stays open until dismissed or timeout

### Command Wake Words (Shortcut Actions)
These skip the general conversation and go straight to a specific tool:

| Wake Word | Action | Session Behavior |
|-----------|--------|-----------------|
| "Hey BodyCam" | Full conversation | Stay in Layer 3 until dismissed |
| "BodyCam, look" | Capture + describe_scene | Quick: capture, describe, speak, вҶ’ back to Layer 2 |
| "BodyCam, read" | Capture + read_text (OCR) | Quick: capture, read, speak, вҶ’ back to Layer 2 |
| "BodyCam, find _____" | Start find_object | Stays in Layer 3 during search |
| "BodyCam, remember" | save_memory (next utterance) | Quick: listen for content, save, вҶ’ back to Layer 2 |
| "BodyCam, translate" | set_translation_mode | Stays in Layer 3 for translation |
| "BodyCam, call _____" | make_phone_call | Quick: dial, вҶ’ back to Layer 2 |
| "BodyCam, navigate" | navigate_to (next utterance) | Stays in Layer 3 for directions |

**Quick actions** connect to the API, execute one tool call, speak the result, then
disconnect and return to Layer 2. This minimizes API cost and battery.

### Sleep Word: "Go to sleep"
- Detected by Porcupine as another wake word
- Transitions from Layer 2 вҶ’ Layer 1
- All listening stops вҖ” complete silence
- Only a physical button press wakes it back to Layer 2
- Visual indicator: LED off or specific color

### Session Dismissal Words (Layer 3 вҶ’ Layer 2)
These are handled by the LLM itself (not Porcupine), since we're already in a
Realtime API session:
- "That's all"
- "Thanks, BodyCam"
- "Go to sleep" (also caught by LLM as intent to end)
- Silence timeout (configurable: 30s - 2 min of no speech)

When dismissed, the orchestrator:
1. Disconnects WebSocket
2. Stops audio streaming
3. Returns to Layer 2 (wake word listening)
4. Plays a short "going quiet" tone

## Porcupine Integration Design

### Why Porcupine

| Criteria | Porcupine | Vosk | Windows SpeechRecognition |
|----------|-----------|------|--------------------------|
| Latency | <50ms | ~200ms | ~300ms |
| Power | ~10mW | ~50mW | ~100mW |
| Custom words | Yes (Console) | No (grammar only) | Yes (constrained) |
| Multiple words | Yes, concurrent | No | Yes |
| Cross-platform | Win/macOS/Linux/RPi | All | Windows only |
| Android | Via MAUI? | Yes | No |
| License | Apache 2.0 | Apache 2.0 | Proprietary |
| Offline | 100% | 100% | 100% |

**Porcupine limitation:** The .NET NuGet (`Porcupine` v4.0.2) supports Windows/macOS/Linux
but **not Android/iOS directly**. For mobile, we'd need `Porcupine.Android` or
`Porcupine.iOS` native packages + platform-specific bindings.

For MVP (Windows desktop/glasses), the .NET package works perfectly.

### Custom Wake Words

Porcupine supports custom wake words via the [Picovoice Console](https://console.picovoice.ai/):
1. Type the desired phrase
2. Platform (Windows/Android/etc.) selected
3. `.ppn` model file generated
4. Ship `.ppn` files as embedded resources

For our needs:
- `hey_bodycam.ppn` вҖ” primary activation
- `bodycam_look.ppn` вҖ” vision shortcut
- `bodycam_read.ppn` вҖ” OCR shortcut
- `bodycam_find.ppn` вҖ” find object shortcut
- `bodycam_remember.ppn` вҖ” memory shortcut
- `bodycam_translate.ppn` вҖ” translation shortcut
- `bodycam_call.ppn` вҖ” phone shortcut
- `bodycam_navigate.ppn` вҖ” navigation shortcut
- `go_to_sleep.ppn` вҖ” sleep command

All loaded concurrently вҖ” Porcupine handles multiple keywords with **no added
runtime cost** (single neural network pass checks all keywords).

## New Service: `IWakeWordService`

```csharp
public interface IWakeWordService
{
    /// <summary>Current listening layer.</summary>
    ListeningLayer CurrentLayer { get; }

    /// <summary>Start wake word detection (enter Layer 2).</summary>
    Task StartAsync(CancellationToken ct = default);

    /// <summary>Stop all listening (enter Layer 1).</summary>
    Task StopAsync();

    /// <summary>Raised when a wake word is detected.</summary>
    event EventHandler<WakeWordDetectedEventArgs> WakeWordDetected;
}

public enum ListeningLayer { Sleep, WakeWord, ActiveSession }

public enum WakeWordType
{
    FullSession,      // "Hey BodyCam"
    Look,             // "BodyCam, look"
    Read,             // "BodyCam, read"
    Find,             // "BodyCam, find"
    Remember,         // "BodyCam, remember"
    Translate,        // "BodyCam, translate"
    Call,             // "BodyCam, call"
    Navigate,         // "BodyCam, navigate"
    GoToSleep         // "Go to sleep"
}

public class WakeWordDetectedEventArgs : EventArgs
{
    public required WakeWordType Type { get; init; }
}
```

### Implementation: `PorcupineWakeWordService`

```csharp
public class PorcupineWakeWordService : IWakeWordService, IDisposable
{
    private readonly IAudioInputService _audioInput;
    private readonly string _accessKey;
    private Porcupine? _porcupine;
    private CancellationTokenSource? _cts;

    public ListeningLayer CurrentLayer { get; private set; } = ListeningLayer.Sleep;
    public event EventHandler<WakeWordDetectedEventArgs>? WakeWordDetected;

    // Keyword paths loaded from embedded resources
    private static readonly string[] KeywordPaths = [
        "hey_bodycam.ppn",
        "bodycam_look.ppn",
        "bodycam_read.ppn",
        "bodycam_find.ppn",
        "bodycam_remember.ppn",
        "bodycam_translate.ppn",
        "bodycam_call.ppn",
        "bodycam_navigate.ppn",
        "go_to_sleep.ppn"
    ];

    // Maps keyword index вҶ’ wake word type (same order as KeywordPaths)
    private static readonly WakeWordType[] KeywordTypes = [
        WakeWordType.FullSession,
        WakeWordType.Look,
        WakeWordType.Read,
        WakeWordType.Find,
        WakeWordType.Remember,
        WakeWordType.Translate,
        WakeWordType.Call,
        WakeWordType.Navigate,
        WakeWordType.GoToSleep
    ];

    public async Task StartAsync(CancellationToken ct = default)
    {
        _porcupine = Porcupine.FromKeywordPaths(_accessKey, KeywordPaths.ToList());
        _cts = new CancellationTokenSource();
        CurrentLayer = ListeningLayer.WakeWord;

        // Start processing audio frames from the mic
        // Porcupine needs 16kHz 16-bit mono PCM, frame size = _porcupine.FrameLength
        _ = Task.Run(() => DetectionLoopAsync(_cts.Token), ct);
    }

    private async Task DetectionLoopAsync(CancellationToken ct)
    {
        while (!ct.IsCancellationRequested)
        {
            var frame = await GetNextAudioFrameAsync(ct); // from IAudioInputService
            var keywordIndex = _porcupine!.Process(frame);

            if (keywordIndex >= 0)
            {
                var type = KeywordTypes[keywordIndex];

                if (type == WakeWordType.GoToSleep)
                {
                    CurrentLayer = ListeningLayer.Sleep;
                    await StopAsync();
                }

                WakeWordDetected?.Invoke(this, new WakeWordDetectedEventArgs { Type = type });
            }
        }
    }

    public Task StopAsync()
    {
        _cts?.Cancel();
        _porcupine?.Dispose();
        _porcupine = null;
        CurrentLayer = ListeningLayer.Sleep;
        return Task.CompletedTask;
    }

    public void Dispose() => StopAsync().GetAwaiter().GetResult();
}
```

## Audio Pipeline Considerations

### Mic Sharing Between Porcupine and Realtime API

The mic can only be captured by one consumer at a time. Two approaches:

**Option A: Exclusive handoff**
```
Layer 2: Mic вҶ’ Porcupine (16kHz, 512-sample frames)
  вҶ“ wake word detected
Layer 3: Stop Porcupine вҶ’ Mic вҶ’ Realtime API (24kHz, PCM16 streaming)
  вҶ“ session ends
Layer 2: Stop streaming вҶ’ Mic вҶ’ Porcupine
```
Simple but has a ~200ms gap during handoff where audio is lost.

**Option B: Shared audio splitter (recommended)**
```
Mic вҶ’ AudioSplitter
  в”ңв”Җв”Җ Porcupine branch (downsample to 16kHz if needed)
  в””в”Җв”Җ Realtime API branch (24kHz, only active in Layer 3)
```
No gap, but requires a small audio routing service. Porcupine can keep running
in Layer 3 to detect "Go to sleep" even during an active session.

**Recommendation:** Start with Option A for simplicity. The 200ms gap is
imperceptible. Move to Option B only if the handoff gap causes problems.

### Sample Rate Mismatch

- Porcupine requires: **16kHz**, 16-bit mono, frame size = 512 samples (32ms)
- Realtime API requires: **24kHz**, 16-bit mono, streaming chunks

The `IAudioInputService` currently captures at 24kHz. Options:
1. Downsample 24kHz вҶ’ 16kHz for Porcupine (simple decimation + anti-alias filter)
2. Switch mic to 16kHz during Layer 2, 24kHz during Layer 3
3. Capture at 48kHz and downsample to both (overkill)

**Recommendation:** Option 1 вҖ” always capture at 24kHz, downsample for Porcupine.

## Risks

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Porcupine false positives | Unwanted API connections ($) | Tune sensitivity (0.5 default); confirmation tone gives user feedback |
| "BodyCam" too common in speech | False triggers | Use "Hey BodyCam" (2-word compound reduces false positive rate 10x) |
| Picovoice AccessKey requirement | Extra setup step for users | Free tier is sufficient; store in settings alongside OpenAI key |
| Mic handoff gap (Option A) | ~200ms of missed audio | Imperceptible; move to Option B if problematic |

| Custom .ppn files per platform | Build complexity | Ship all platform variants as embedded resources |

## Cost Comparison

| Mode | Hourly cost (audio input) | Battery draw | Use case |
|------|--------------------------|-------------|----------|
| Always-on Realtime API | ~$5.76/hr | ~300mW | Never вҖ” too expensive |
| Wake word + quick actions | ~$0.10-0.50/hr | ~10mW idle, ~300mW burst | Recommended |
| Push-to-talk only | ~$0.05-0.20/hr | ~0mW idle, ~300mW burst | Fallback if no Porcupine key |

Wake word mode reduces cost by **10-50x** vs. always-on, with minimal UX degradation.
The user just says "Hey BodyCam" before speaking вҖ” same as every smart assistant.
