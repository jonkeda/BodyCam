# RCA-804: BLE SSID Extraction — GATT Service Mapping & Protocol Findings

**Severity:** Informational — research results from Option D (RCA-803)  
**Component:** `WindowsHeyCyanGlassesSession` BLE communication  
**Platform:** Windows (unpackaged / `WindowsPackageType=None`)  
**Date:** 2026-05-17  
**Device:** HeyCyan M01 Pro (MAC D8:79:B8:7F:E6:C9)  
**Firmware:** AM01C_2.00.03_250718  
**Hardware:** AM01C_V2.0  

---

## Context

RCA-803 concluded that all WinRT WiFi Direct discovery approaches fail on
Windows and identified "BLE SSID extraction" (Option D) as the best path.
The hypothesis: the iOS QCSDK sends opcode `0x41` (SetDeviceMode) + mode
`0x04` (Transfer) via BLE, and the glasses respond with SSID + password on
the notify characteristic.

This RCA documents the systematic testing of that hypothesis.

---

## Experiment 1: `{0x41, 0x04}` on All Writable Characteristics

Sent the iOS-style opcode `{0x41, 0x04}` to every writable characteristic
while subscribed to ALL notify/indicate characteristics across all services.

| Target Char | Service | Write Result | Response |
|---|---|---|---|
| **ae01** | ae30 | Success | **ae02: `41-04` (echo)** |
| ae03 | ae30 | Success | No response on ae04 |
| ae10 | ae30 | Success | No response |
| 4a02 | 3802 | Success | No response |
| ae3b | ae3a | Success | No response |
| 6e400002 | NUS | Success | No response |
| de5bf72a | Serial | Success | No response |
| fee3 | fee1 | Success | No response |
| 2a00 | GAP | AccessDenied | — |

**Result:** Only ae01→ae02 responded, with a verbatim echo. No SSID data.

## Experiment 2: GAIA-Formatted Commands on ae01

Tested whether `0000ae30` is a Qualcomm GAIA service by sending
properly formatted GAIA v3 BLE frames (Vendor ID + Command ID + Payload):

| Command Format | Bytes | ae02 Response |
|---|---|---|
| Raw `{0x41, 0x04}` | `41-04` | Echo: `41-04` |
| GAIA V=001D C=0041 P=04 | `00-1D-00-41-04` | Echo: `00-1D-00-41-04` |
| GAIA V=000A C=0041 P=04 | `00-0A-00-41-04` | Echo: `00-0A-00-41-04` |
| GAIA V=001D C=0104 | `00-1D-01-04` | Echo: `00-1D-01-04` |
| GAIA V=001D C=0201 P=04 | `00-1D-02-01-04` | Echo: `00-1D-02-01-04` |
| Serial via GAIA `{02,01,04}` | `02-01-04` | Echo: `02-01-04` |
| GAIA V=00D2 C=0041 P=04 | `00-D2-00-41-04` | Echo: `00-D2-00-41-04` |
| GAIA V=001D C=4104 | `00-1D-41-04` | Echo: `00-1D-41-04` |

**Every response was a verbatim echo.** The ae30 service is a simple
loopback channel, not a GAIA command processor.

## Experiment 3: Serial Port Notification Health

During the GAIA echo tests, the serial port (de5bf729) delivered two
battery notifications:

```
BC-73-03-00-7F-C1-05-28-00   → battery 40% (type 0x05)
BC-73-03-00-7A-31-05-27-00   → battery 39% (type 0x05)
```

This confirms:
- ✅ BLE notification pipeline is functional
- ✅ Glasses are sending periodic data on de5bf729
- ❌ EnterTransferMode (`{0x02, 0x01, 0x04}`) produces NO mode-change ack (0x01)
  or transfer IP (0x08) in the current session

In a prior session (same day, earlier), EnterTransferMode DID produce a
0x01 mode-change ack: `BC-73-08-00-9C-C7-01-0D-00-00-00-00-00-01`. The
inconsistency may be battery-related (52% earlier → 39% now) or state-related
(glasses may enter a degraded mode at low battery).

---

## Experiment 4: Fresh EnterTransferMode After State Reset

Sequence: ExitTransferMode → wait 3s → ResetP2p → wait 3s → GetBattery →
EnterTransferMode → wait 20s.

**GetBattery:** Returned 0% (timeout — no 0x05 response received within 3s).

**EnterTransferMode:** Zero notifications received in 20 seconds.

The glasses were becoming unresponsive at this point (39% battery,
multiple test sessions). Subsequent connection attempts failed with
`TaskCanceledException`.

## Conclusions

### What we know

1. **The `0x41` opcode is not recognized** by the glasses on any of 9 GATT
   services. The iOS QCSDK `ODM_DFU_Operation_SetDeviceMode` protocol is
   not exposed via any discoverable BLE characteristic.

2. **Service `0000ae30` is a loopback echo**, not Qualcomm GAIA. It echoes
   any bytes written to ae01 back on ae02. Characteristics ae03/ae04 accept
   writes silently but produce no response.

3. **The serial port channel works** (battery notifications flow), but
   EnterTransferMode stopped producing mode-change acks during this session.
   This may be transient (battery/state dependent).

4. **The iOS SDK's `openWifiWithMode:`** likely uses a protocol detail we
   haven't discovered — possibly a multi-step handshake, a different command
   encoding, or interaction with a characteristic we didn't test correctly
   (ae03 with GAIA framing was untested due to connection loss).

5. **Glasses device name "M01 Pro_E6C9"** contains BLE MAC suffix and
   starts with `M01` — matching our SSID filters. The WiFi Direct SSID
   may follow a predictable pattern like `DIRECT-xx-M01 Pro_E6C9`.

### What we don't know

1. Exact command format the iOS SDK sends (binary framework, not in headers)
2. Whether ae03/ae04 is the real command channel (untested with GAIA framing)
3. Whether the glasses WiFi Direct SSID follows a predictable naming pattern
4. Why EnterTransferMode ack stopped working mid-session

## Recommended Next Steps

### 1. BLE traffic capture (highest value)

Use nRF Connect on an Android phone to sniff BLE traffic while an iOS
device runs the QCSDK demo app's `openWifiWithMode:` call. This reveals
the exact:
- BLE service + characteristic used
- Command byte sequence
- Response format containing SSID + password
- Any multi-step handshake

### 2. QCSDK binary disassembly

Disassemble `QCSDK.framework/QCSDK` (arm64 Mach-O) with Ghidra or Hopper.
Search for:
- References to `ODM_DFU_Operation_SetDeviceMode`
- Characteristic UUID constants (`de5bf72a`, `ae01`, etc.)
- String literals containing "SSID", "password", "DIRECT"

### 3. Known SSID brute-force (low effort)

After entering transfer mode via `{0x02, 0x01, 0x04}` on de5bf72a,
try connecting to predictable SSIDs with password `"123456789"`:
- `DIRECT-M01 Pro_E6C9`
- `DIRECT-xx-M01 Pro_E6C9` (xx = random 2 chars)
- `M01_Pro_E6C9`
- `DIRECT-E6C9`

Use `netsh wlan connect` or `WiFiAdapter.ConnectAsync()` to attempt joining.

### 4. Retry ae03 with GAIA framing (when glasses charged)

ae03→ae04 accepted writes silently (no echo like ae01). This could be the
real command channel. Try:
- `{0x00, 0x1D, 0x00, 0x41, 0x04}` on ae03 (GAIA V=001D, C=0041, P=04)
- Other GAIA-formatted commands
- Wait longer (30s) for ae04 response

### 5. Re-verify EnterTransferMode with fresh battery

The mode-change ack DID work earlier in the session. Retest after full
charge to confirm it's not a firmware regression.
