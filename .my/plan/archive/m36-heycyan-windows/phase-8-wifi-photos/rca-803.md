# RCA-803: WiFi Direct Peer Discovery Fails on Windows

**Severity:** Critical — blocks all photo transfer from glasses on Windows  
**Component:** `WindowsWiFiDirectManager` → WinRT `DeviceInformation.CreateWatcher`  
**Platform:** Windows (unpackaged / `WindowsPackageType=None`)  
**First seen:** 2026-05-17  
**Confirmed by:** Real-hardware test `TransferMode_TakePhotoThenDownload_RoundTrip`

---

## Symptom

After sending BLE `EnterTransferMode` (`02 01 04`), the glasses start a
WiFi Direct P2P group. The Windows `DeviceInformation.CreateWatcher` with
`WiFiDirectDeviceSelectorType.AssociationEndpoint` finds other WiFi Direct
devices (Samsung TV) but **never discovers the glasses peer**. All four
IP resolution strategies fail, and `EnterTransferModeAsync` throws
`TimeoutException` after 90 seconds.

```
[WIFIDIRECT] Peer found: 'Samsung AU7040 50 TV'
[WIFIDIRECT] Failed: A task was canceled.
[WIFI] No glasses hotspot found. Visible SSIDs (15): [...]
[TRANSFER] Waiting for BLE notify (0x08 or 0x01) as last resort...
System.TimeoutException: No response for notify type 0x01 within 90s
```

---

## Root Cause

The Windows implementation uses a fundamentally different P2P discovery
mechanism than Android, missing several critical protocol details.

### Android (working) vs Windows (broken)

| Aspect | Android (CyanBridge) | Windows (current) |
|--------|---------------------|-------------------|
| **Discovery API** | `WifiP2pManager.discoverPeers()` — P2P-specific active scan | `DeviceInformation.CreateWatcher` — passive WinRT device enumeration |
| **WPS mode** | Explicit `wps.setup = 0` (Push Button Config — no credentials) | Not configured — `FromIdAsync()` may attempt unknown pairing |
| **Connection** | `wifiP2pManager.connect(channel, config, listener)` — P2P connect with WPS PBC | `WiFiDirectDevice.FromIdAsync(deviceId)` — generic device open |
| **IP resolution** | `WifiP2pInfo.groupOwnerAddress` from broadcast callback | `GetConnectionEndpointPairs()[0].RemoteHostName` |
| **Network binding** | `ConnectivityManager.bindProcessToNetwork()` — forces HTTP over P2P | Not done (Windows may auto-route, or may not) |
| **Discovery timeout** | 16s, then retry with glasses P2P reset | 45s hard timeout, no retry |
| **Connect timeout** | 5s, 1 retry | No separate connect timeout |
| **Retry logic** | 1 discovery retry + 1 connect retry + reconnect on disconnect | Single attempt, no retry |
| **GroupOwnerIntent** | Default (-1, framework negotiates; glasses become GO) | Not applicable (never reaches connection) |

### Three contributing factors

#### 1. Wrong discovery mechanism (primary cause)

`DeviceInformation.CreateWatcher` with `WiFiDirectDeviceSelectorType.AssociationEndpoint`
performs passive device enumeration. It finds persistently-advertised WiFi Direct
devices (like the Samsung TV) but may not detect **transient P2P group owners**
that only advertise briefly after a BLE trigger.

Android's `WifiP2pManager.discoverPeers()` is an **active P2P scan** that sends
probe requests on the P2P discovery channel (social channels 1, 6, 11). This is
fundamentally different from WinRT's device watcher.

WinRT has a closer equivalent: `WiFiDirectAdvertisementPublisher` and
`WiFiDirectDevice` with explicit connection parameters. The current code
doesn't use these.

#### 2. Missing WPS Push Button Configuration

The Android code explicitly sets `wps.setup = 0` (WPS PBC) — no PIN, no
password. This is how the glasses expect to be paired. The Windows code
calls `WiFiDirectDevice.FromIdAsync(deviceId)` with no connection parameters,
which may attempt a different pairing method or prompt a user consent dialog
that's never answered in the test.

The WinRT equivalent would be:
```csharp
var parameters = new WiFiDirectConnectionParameters();
parameters.PreferredPairingProcedure = WiFiDirectPairingProcedure.GroupOwnerNegotiation;
// or: parameters.PreferredPairingProcedure = WiFiDirectPairingProcedure.Invitation;

var device = await WiFiDirectDevice.FromIdAsync(deviceId, parameters);
```

#### 3. Unpackaged app capability gap (possible)

The `wifiDirect` device capability is declared in `Package.appxmanifest`:
```xml
<DeviceCapability Name="wifiDirect" />
```

But the app runs as unpackaged (`WindowsPackageType=None`). WinRT capabilities
declared in the manifest may not be enforced for unpackaged apps — the capability
model works differently. This could silently prevent WiFi Direct discovery
from returning any P2P peers, while still allowing pre-paired devices (like the
Samsung TV) to appear.

## Fix Options

### ~~Option A — Use WinRT WiFi Direct active discovery~~ RULED OUT

All three WinRT selector types were tested in diagnostic run (2026-05-17):

| Selector | Peers Found | Glasses? |
|----------|-------------|----------|
| `AssociationEndpoint` | 1 (Samsung TV) | No |
| `DeviceInterface` | 0 | No |
| `Default` (= DeviceInterface) | 0 | No |
| `FindAllAsync` snapshots (×4) | 1 (Samsung TV) | No |
| Direct `FromIdAsync` by BLE MAC ±0/±1/±2 | COMException ×5 | No |

**Conclusion:** WinRT WiFi Direct APIs cannot discover the glasses P2P group
at all. The glasses' transient P2P advertisement is invisible to all three
selector types, and direct connection by constructed device ID fails.

### ~~Option B — Native wlanapi.dll WiFi Direct (WFD* APIs)~~ RULED OUT

Research confirmed the native `wlanapi.dll` WFD functions (`WFDOpenHandle`,
`WFDStartOpenSession`, `WFDOpenLegacySession`, `WFDUpdateDeviceVisibility`)
**all require prior Windows Pairing experience**. From Microsoft docs:

> "For a desktop app, the Wi-Fi Direct feature requires that Wi-Fi Direct
> devices be previously paired by the user with the Windows Pairing
> experience user interface."

`WFDStartOpenSession` accepts a raw MAC address, bypassing WinRT
`DeviceInformation`, but the device must already have a Device Node from
pairing. There is **no active P2P scan** equivalent to Android's
`discoverPeers()` in the native API. The native API is strictly a subset
of WinRT for post-pairing connection establishment.

**NuGet package:** `Vanara.PInvoke.WlanApi` (v5.0.5, MIT, excellent quality)
provides complete P/Invoke wrappers for all WFD* functions — but the API
limitation makes this approach unviable.

### ~~Option C — WiFi hotspot SSID scan~~ RULED OUT

WiFi SSID diff test (6 scans over 30 seconds after transfer mode entry)
found **zero new SSIDs**. No `DIRECT-xx-...` prefixed networks appeared.
The glasses' P2P group is invisible to regular WiFi scans on Windows.

IP brute-force probing (192.168.43.1, .4.1, .31.1, etc.) returned only
the home router (192.168.1.1) — no glasses HTTP server reachable.

### Option D — Reverse-engineer QCSDK BLE protocol for SSID ⭐ BEST PATH

The iOS SDK (`QCSDK.framework`) extracts SSID + password from the BLE
response to `openWifiWithMode:QCOperatorDeviceModeTransfer`. The protocol:

1. **Write** to BLE command characteristic:
   opcode `ODM_DFU_Operation_SetDeviceMode` (`0x41`) + mode `0x04` (Transfer)
2. **Receive** notify response containing SSID + password strings
3. **Join** the glasses WiFi Direct group as a regular network using the SSID/password
   (iOS does this via `NEHotspotConfiguration(ssid, password)`)

**What we know:**
- The BLE command characteristic is `de5bf72a-d711-4e47-af26-65e3012a5dc7` (already used)
- The notify characteristic is `de5bf729-d711-4e47-af26-65e3012a5dc7` (already subscribed)
- The current `EnterTransferMode` sends `{0x02, 0x01, 0x04}` — this is the
  `HeyCyanCommands` format, NOT the `0x41` opcode format from the iOS SDK
- The iOS SDK may use a **different command format** with opcode `0x41`
- The glasses already return a 14-byte 0x01 ack — there may be a second response
  frame containing SSID/password that we're not triggering or not parsing

**Action plan:**
1. Try sending `{0x41, 0x04}` (SetDeviceMode + Transfer) instead of `{0x02, 0x01, 0x04}`
2. Log ALL BLE notify frames after the command to find SSID/password data
3. If SSID is obtained, join via `WiFiAdapter.ConnectAsync(network, credential)`
4. The password is likely `"123456789"` (hardcoded in iOS `GlassesWiFiHandler`)

**NuGet support:**
- `Vanara.PInvoke.WlanApi` — for lower-level WiFi control if `WiFiAdapter` is insufficient
- `ManagedNativeWifi` (v3.0.2, MIT) — simpler WiFi connect/scan API

### Option E — Android relay (backup plan)

Use the Android phone (which has working WiFi Direct) as a relay:
1. Android app downloads photos via its working P2P connection
2. Relays photo bytes to Windows over BLE GATT, TCP socket, or HTTP
3. Requires implementing a relay service in the Android app

**Pros:** Works without solving the Windows WiFi Direct problem.
**Cons:** Requires Android phone, adds latency, complex architecture.

|---|---|---|---|

| ae03 | ae30 | Success | No response on ae04 |
| ae10 | ae30 | Success | No response |

processing.** The glasses echo any bytes written to ae01 back on ae02.

| Command | Bytes | ae02 Response |
|---|---|---|
| Raw `{0x41, 0x04}` | `41-04` | Echo: `41-04` |
| GAIA V=001D C=0041 P=04 | `00-1D-00-41-04` | Echo: `00-1D-00-41-04` |
| GAIA V=000A C=0041 P=04 | `00-0A-00-41-04` | Echo: `00-0A-00-41-04` |
| GAIA V=001D C=0104 | `00-1D-01-04` | Echo: `00-1D-01-04` |
| GAIA V=001D C=0201 P=04 | `00-1D-02-01-04` | Echo: `00-1D-02-01-04` |
| Serial cmd via GAIA | `02-01-04` | Echo: `02-01-04` |
| GAIA V=00D2 C=0041 P=04 | `00-D2-00-41-04` | Echo: `00-D2-00-41-04` |

**All echoed verbatim.** ae01→ae02 is a loopback, no GAIA processing.

**However**, two serial port battery notifications (0x05) DID appear during
this test on de5bf729:
- `BC-73-03-00-7F-C1-05-28-00` (battery 40%)
- `BC-73-03-00-7A-31-05-27-00` (battery 39%)

This confirms the serial port notification pipeline works. The EnterTransferMode
command is simply not producing mode-change ack responses in the current session.

## Recommended Next Step

~~**Option D** — BLE SSID extraction via `{0x41, 0x04}` opcode~~ **RULED OUT**

The `0x41` opcode produces no meaningful response on any of the 9 GATT
services. The ae30 service is a simple echo/loopback, not a GAIA command
processor. The glasses firmware (AM01C_2.00.03_250718) does not appear to
support the iOS QCSDK `SetDeviceMode` protocol over any exposed BLE
characteristic.

### Remaining viable approaches

1. **BLE traffic sniffing** — Use an iOS device with the QCSDK demo app
   and a BLE sniffer (nRF Connect, Wireshark + Nordic dongle) to capture
   the exact BLE command sequence that the iOS SDK sends to `openWifiWithMode`.
   This would reveal the actual command format and characteristic used.

2. **Binary analysis of QCSDK.framework** — Disassemble the iOS SDK binary
   to extract the exact BLE write sequence for `openWifiWithMode:`. The
   binary is an arm64 Mach-O; tools like Hopper, Ghidra, or `otool` can
   extract the command bytes.

3. **Known SSID pattern** — The glasses device name is "M01 Pro_E6C9".
   If the WiFi Direct SSID follows a predictable pattern (e.g., `DIRECT-xx-M01 Pro_E6C9`),
   we could construct the SSID and try joining with the known password
   `"123456789"` without needing BLE extraction.

4. **Android relay** — Use the Android phone (which has working WiFi Direct
   P2P via `discoverPeers()`) as a relay for photo download.
