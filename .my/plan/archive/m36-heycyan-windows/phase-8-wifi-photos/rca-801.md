# RCA-601: Preferences.dat File Contention on Windows Unpackaged

**Severity:** High — crashes app during startup and settings save  
**Component:** `SettingsService` → `Microsoft.Maui.Storage.Preferences`  
**Platform:** Windows (unpackaged / `WindowsPackageType=None`)  
**First seen:** 2026-05-17

---

## Symptom

```
System.IO.IOException: The process cannot access the file
  'C:\Users\...\com.companyname.bodycam\Settings\preferences.dat'
  because it is being used by another process.
```

Stack trace:
```
SafeFileHandle.CreateFile(...)
SafeFileHandle.Open(... FileShare share ...)
OSFileStreamStrategy..ctor(...)
File.Create(path)                              ← FileShare.None
UnpackagedPreferencesImplementation.Save()
UnpackagedPreferencesImplementation.Set[T](key, value, sharedName)
```

## Root Cause

**MAUI's `UnpackagedPreferencesImplementation.Save()` opens `preferences.dat`
with `FileShare.None` (exclusive lock).** When two `Preferences.Set()` calls
execute concurrently, the second call fails because the first still holds the
file lock.

`SettingsService` wraps every property as a direct `Preferences.Set()` call
with no synchronization:

```csharp
public string RealtimeModel
{
    get => Preferences.Get(nameof(RealtimeModel), ...);
    set => Preferences.Set(nameof(RealtimeModel), value);  // ← direct file I/O
}
```

### Trigger points (concurrent writes)

| Location | Writes | Risk |
|----------|--------|------|
| `MauiProgram.cs` startup — loading .env Azure config | 6 rapid sequential | **HIGH** — main thread, synchronous |
| `ConnectionViewModel` — user changes provider + endpoint | 2–5 from UI thread | **MEDIUM** — user-driven |
| `SetupViewModel.ValidateAndSaveAsync` | 2 writes in async | **MEDIUM** |
| BLE reconnect → `LastHeyCyanDeviceAddress` + `LastHeyCyanDeviceName` | 2 writes from BT callback | **HIGH** — background thread |

The 6 rapid writes in `MauiProgram.cs` are the most likely trigger — each
`Preferences.Set()` call does a full serialize-to-disk, and the file may
still be locked by the OS when the next call executes.

### Why this only affects Windows unpackaged

- **Android/iOS:** `Preferences` uses `SharedPreferences` / `NSUserDefaults`
  (in-memory with deferred flush) — no file contention possible.
- **Windows packaged (MSIX):** Uses `ApplicationData.Current.LocalSettings`
  (registry-backed) — no file contention.
- **Windows unpackaged:** Uses `UnpackagedPreferencesImplementation` which
  serializes to a flat file with exclusive file locking on every write.

---

## Fix

Wrap `Preferences.Set()` calls with a process-wide lock to serialize file
access. The lock only needs to protect writes — reads are safe because
`UnpackagedPreferencesImplementation` caches values in memory.

### Option A — Lock in SettingsService (minimal change)

```csharp
public class SettingsService : ISettingsService
{
    private static readonly object _prefsLock = new();

    private void SetPref<T>(string key, T value)
    {
        lock (_prefsLock)
        {
            Preferences.Set(key, value);
        }
    }

    public string RealtimeModel
    {
        get => Preferences.Get(nameof(RealtimeModel), ...);
        set => SetPref(nameof(RealtimeModel), value);
    }
    // ... all other setters use SetPref
}
```

### Option B — Batch writes at startup

Collect all startup settings into a single `Preferences.Set()` sequence
with a lock, or defer them entirely (since `AppSettings` already holds
runtime copies).

### Option C — Replace Preferences with a custom JSON store

Use a single JSON file with `FileShare.Read` and a write lock. This avoids
MAUI's broken file handling entirely.

---

## Recommendation

**Option A** — quickest fix, addresses the root cause without changing
architecture. Add a `static readonly object _prefsLock` and route all
setters through a locked helper.

---

## Validation

1. Rapid sequential writes: call 10+ property setters in a tight loop
   from a unit test — should not throw.
2. Concurrent writes: spawn 5 threads each writing different properties
   simultaneously — should not throw.
3. Startup path: run the app with `.env` Azure config present — no crash.
