# RCA-802: Test Capture Image Not Displayed After Successful Capture

**Severity:** Medium — captured image not visible to user  
**Component:** `GlassesCameraSectionViewModel` → `HeyCyanCameraProvider` → `HeyCyanMediaTransfer`  
**Platform:** Windows (HeyCyan glasses via WiFi Direct)  
**First seen:** 2026-05-17

---

## Symptom

The user presses "Test Capture" on the Device Settings page (Camera Source
section). The button invokes the capture pipeline, but no image appears
below the button. It is unclear whether:

1. The capture succeeded but the image failed to render, or
2. The capture itself failed silently.

The UI shows "Camera Source: HeyCyan Glasses Camera" and the Test Capture
button is enabled (meaning `HeyCyanCameraProvider.IsAvailable` is `true`,
i.e. the session is in `Connected` or `TransferMode` state).

---

## Root Cause Analysis

There are **multiple failure points** in the pipeline from button press to
image display. Any one of them silently swallows the error and results in
no image shown:

### 1. Transfer mode entry failure (most likely)

`CaptureFrameAsync` step 1 calls `_transfer.ListAsync()`, which calls
`EnsureTransferModeAsync()`. This triggers the full WiFi Direct connection
sequence:

```
BLE ResetP2P → BLE EnterTransferMode → WiFi Direct discovery → peer connect → HTTP
```

**Known failure modes on Windows:**
- WiFi Direct discovery finds no peers (glasses P2P advertisement missed)
- WiFi Direct `FromIdAsync()` fails (unpackaged app lacks `wifiDirect` capability)
- BLE 0x08 IP notify never arrives (only comes after WiFi Direct connects)
- All 4 strategies fail → `InvalidOperationException` thrown

The exception is caught by `CaptureFrameAsync`'s outer catch:
```csharp
catch (Exception ex)
{
    _log.LogError(ex, "HeyCyan photo capture failed");
    return null;  // ← swallowed, no re-throw
}
```

Then in `TestCaptureAsync`:
```csharp
if (jpg is null)
{
    Status = "Error: Capture returned null";
    LastCaptureImage = null;       // ← image stays invisible
    LastCaptureLatencyMs = null;   // ← latency stays invisible
}
```

**The user sees "Error: Capture returned null" in the status label but no
image.** However, the status label may not be prominently visible, and the
user may not notice it — especially since the button area returns to its
default state.

### 2. BLE photo command sent without transfer mode

`CaptureFrameAsync` step 2 calls `_session.TakePhotoAsync()`, which sends
`HeyCyanCommands.StartPhotoMode()` via BLE. This tells the glasses to take
a photo — but the glasses store it locally. To download the photo, the
transfer session must already be warm (HTTP connection active). The flow is:

```
1. ListAsync() → enters transfer mode (WiFi Direct + HTTP)
2. TakePhotoAsync() → BLE command to take photo
3. WaitForNewPhoto() → wait for media count notification
4. DownloadAsync() → download via HTTP
```

If step 1 fails, step 2 still executes (BLE command succeeds independently
of WiFi), but steps 3-4 will also fail because there's no HTTP connection.

### 3. `WaitForNewPhotoAsync` timeout (6 seconds)

After taking the photo, the code waits 6 seconds for a `MediaCountUpdated`
event. If the glasses are slow or the BLE notification is missed, this times
out and falls through to the fallback `ListAsync()` call — which may also
fail if the transfer session tore down during the wait (8-second warm idle
timeout in `HeyCyanMediaTransfer`).

### 4. `ImageSource.FromStream` MAUI rendering issue (unlikely but possible)

If the capture succeeds and returns valid JPEG bytes:
```csharp
LastCaptureImage = ImageSource.FromStream(() => new MemoryStream(jpg));
```

MAUI's `StreamImageSource` evaluates the factory lambda when the `Image`
control measures/draws. On Windows (WinUI3), there are known issues where
`StreamImageSource` fails silently if:
- The stream is consumed before the layout pass
- The image control is inside a `VerticalStackLayout` with no explicit width

This is less likely given the code creates a fresh `MemoryStream` per call.

## Diagnostic Gaps

The current code makes it hard to tell what happened:

| What we need | What we have |
|---|---|
| Whether WiFi Direct connected | Console.Error.WriteLine only (not in UI) |
| Whether transfer mode entered | Log only, Status shows "Capturing…" during |
| Whether TakePhoto BLE cmd was sent | Log only |
| Whether media count notify arrived | Log only |
| Whether HTTP download succeeded | Log only |
| The actual error message | `Status = "Error: Capture returned null"` — no detail |
| Whether the image bytes are valid | `AssertJpegMagic` throws, caught by outer catch |

## Proposed Fix

### A. Surface the actual error to the UI (quick win)

Change `CaptureFrameAsync` to propagate the exception instead of returning
null, so `TestCaptureAsync` can show the real error:

```csharp
// In TestCaptureAsync:
catch (Exception ex)
{
    Status = $"Error: {ex.Message}";           // ← shows the REAL failure
    LastCaptureImage = null;
    LastCaptureLatencyMs = null;
    _log.LogError(ex, "Test capture failed");
}
```

### B. Add progress feedback during transfer mode entry

The transfer mode entry can take 10-45 seconds (WiFi Direct discovery +
connect). The UI only shows "Capturing…" with no progress. Add intermediate
status updates:

```
"Entering transfer mode…"
"WiFi Direct: discovering peers…"
"WiFi Direct: connecting…"
"Taking photo…"
"Downloading…"
"Done: 245,832 bytes in 12,340 ms"
```

This requires either a progress callback from `HeyCyanCameraProvider` or
splitting the status updates into the ViewModel.

### C. Verify WiFi Direct works in unpackaged mode

The `wifiDirect` DeviceCapability is declared in `Package.appxmanifest`,
but the app runs as unpackaged (`WindowsPackageType=None`). WinRT WiFi
Direct APIs may require a packaged app or explicit capability declaration
via other means. If WiFi Direct silently fails due to capability issues,
the entire capture pipeline breaks at step 1.

Test: `TransferMode_TakePhotoThenDownload_RoundTrip` — **FAILED** after 1m 33s.

### Timeline

| Time | Event | Detail |
|------|-------|--------|
| 0s | BLE connect | Battery, time sync OK |
| ~1s | WiFi Direct discovery starts | Selector: `AssociationEndpoint` + `WiFiDirect` protocol |
| ~2s | Peer found: Samsung TV | `Samsung AU7040 50 TV` (not glasses) |
| ~2s | BLE: `ResetP2p` sent | `02-01-0F` |
| ~3s | BLE: `EnterTransferMode` sent | `02-01-04` |
| ~3s | BLE 0x01 notify received | `BC-73-08-00-9C-C7-01-0D-00-00-00-00-00-01` — mode-change ack, no IP |
| ~3-48s | WiFi Direct: waiting for glasses peer | **No glasses peer ever appears** |
| ~48s | WiFi Direct timeout | `A task was canceled` (45s CTS) |
| ~51s | WiFi scan #1 | 15 SSIDs, no glasses hotspot |
| ~54s | WiFi scan #2 | 13 SSIDs, no glasses hotspot |
| ~57s | WiFi scan #3 | 15 SSIDs, no glasses hotspot |
| ~57s | Last resort: wait for BLE 0x01 or 0x08 | 0x01 already consumed, 0x08 never arrives |
| 93s | **TimeoutException** | `No response for notify type 0x01 within 90s` |

### Key observations

1. **The glasses never appear as a WiFi Direct peer.** Only a Samsung TV was
   discovered. The glasses P2P advertisement is either:
   - Not being emitted (glasses firmware issue or transfer mode not fully activated)
   - Emitted but invisible to WinRT `DeviceInformation.CreateWatcher` (API limitation)
   - Emitted with a device name that doesn't match `IsLikelyGlassesPeer()`

2. **The BLE 0x01 frame arrives immediately** (within ~1s of `EnterTransferMode`),
   but it's a mode-change acknowledgment with payload `0D-00-00-00-00-00-01` —
   no IP address. The code logs it during Strategy 2b but can't extract an IP.

3. **The BLE 0x08 IP notify never arrives.** On Android, this only comes AFTER
   the WiFi Direct P2P connection is established. Since Windows never connects
   P2P, the glasses never send the IP.

4. **WiFi hotspot fallback finds nothing.** The glasses use WiFi Direct, not
   a regular hotspot (confirmed by protocol analysis). Regular WiFi scans
   will never find them.

5. **Chicken-and-egg problem:** Windows needs the glasses IP to connect HTTP,
   but the glasses only send the IP (0x08) after a WiFi Direct peer connects.
   Windows can't connect WiFi Direct because it can't find the glasses peer.

### Next steps

- **Investigate why WinRT WiFi Direct watcher doesn't find the glasses:**
  - Try `WiFiDirectDeviceSelectorType.DeviceInterface` instead of `AssociationEndpoint`
  - Check if the glasses require WPS PBC (Push Button Configuration) pairing
  - Check if `WiFiDirectDevice.FromIdAsync` needs `WiFiDirectConnectionParameters`
    with `GroupOwnerIntent = 0` (let glasses be GO)
  - Try running as Administrator or as a packaged app
- **Android reference:** Android uses `wifiP2pManager.discoverPeers()` + broadcast
  receiver, which is a different API surface than WinRT `DeviceInformation.CreateWatcher`
