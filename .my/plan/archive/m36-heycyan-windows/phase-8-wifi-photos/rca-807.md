# RCA-807: Windows BLE GATT Notifications Silently Drop After Initial Session

## Status: CONFIRMED — Device Firmware Disabled All Notifications (Low Battery Power-Saving)

## Problem Statement

BLE notification delivery from HeyCyan glasses (`de5bf729` characteristic) is **intermittent and unreliable** on Windows. The `GattCharacteristic.ValueChanged` event stops firing despite:

- CCCD write returning `GattCommunicationStatus.Success`
- BLE writes to the TX characteristic succeeding (confirmed by device behavior)
- Connection status remaining `Connected`
- `GattSession.MaintainConnection = true` being set
- Power cycling the glasses between test sessions

This blocks all command-response flows that depend on BLE notifications, including `openWifiWithMode` WiFi credential retrieval, battery queries, and mode-change confirmations.

## Evidence

### Session 2025-05-18 Timeline

| # | Test | Notifications Received | Notes |
|---|------|----------------------|-------|
| 1 | `DiagBleSetDeviceMode_CaptureSSIDFromNotify` | **1** (14-byte ACK) | `{0x41, 0x04}` → `BC-73-08-00-41-06-01-11-00-00-00-00-00-01` |
| 2 | SSID diff test (WiFi scan) | N/A | No BLE notifications expected |
| 3 | `Rca806_PhaseC` (first run) | **0** | Baseline `{0x41, 0x04}` failed, all 13 hypotheses = 0 |
| 4 | *Power cycle glasses* | — | Hold 5s off, wait 10s, hold 3s on |
| 5 | `Rca806_PhaseC` (second run) | **0** | Same baseline `{0x41, 0x04}` = 0 even after full power cycle |

Key observation: The **exact same command** (`{0x41, 0x04}`) that produced a notification in test #1 produced **zero** notifications in tests #3 and #5.

### What `[BLE-NOTIFY]` Absence Proves

The notification handler in `WindowsHeyCyanGlassesSession.OnCharacteristicValueChanged` writes to stderr:
```csharp
Console.Error.WriteLine($"[BLE-NOTIFY] ({bytes.Length}B) {BitConverter.ToString(bytes)}");
```

Test output shows `[BLE-WRITE]` lines for every command sent, but **zero** `[BLE-NOTIFY]` lines. This proves `GattCharacteristic.ValueChanged` is simply not being raised by the Windows BLE stack.

## Root Cause Analysis

### What We Can Rule Out

1. **Command format wrong** — The same `{0x41, 0x04}` command that failed later DID produce a notification earlier in the same session
2. **Glasses firmware bug** — Power cycling resets all glasses state; issue persists
3. **CCCD write failure** — Returns `Success` (if it failed, ConnectAsync would throw)
4. **Connection dropped** — Writes succeed, state reports `Connected`
5. **Event handler unsubscribed** — Same code path in all tests; handler is subscribed before any writes

### Most Likely Cause: Device Firmware Power-Saving Mode

**CONFIRMED 2025-07-27**: Comprehensive all-service diagnostic subscribed to ALL 8 notify/indicate
characteristics across all 9 GATT services. Waited 180 seconds. **ZERO notifications from ANY
characteristic.** This is NOT a Windows BLE stack issue — the glasses' firmware has stopped sending
all BLE notifications.

Evidence timeline:
- RCA-804 (earlier): Battery notifications arrived spontaneously at 52% battery
- RCA-804 (later): Notifications stopped at 39% battery
- RCA-807: After 50+ connection/test cycles, no notifications from any of 8 subscribed characteristics

Characteristics subscribed (all returned CCCD Success):
- `0000ae02` (ae30 echo service) — Notify
- `0000ae04` (ae30 service) — Notify
- `0000ae05` (ae30 service) — Indicate
- `00004a02` (3802 service) — Notify
- `0000ae3c` (ae3a service) — Notify
- `6e400003` (NUS) — Notify
- `de5bf729` (serial port) — Notify
- `0000fee3` (fee1 service) — Notify

The connection remained healthy (`Connected` status, MaxPdu=517), writes succeeded (EnterTransferMode
physically changed device mode), but the device firmware simply does not produce any notification PDUs.

**Root cause**: The glasses enter a low-battery power-saving state that disables all notification
sending. Writes are still accepted (they're lightweight), but the firmware stops all outbound
notification/indication traffic to conserve power.

**Resolution**: Fully charge the glasses (case for 1-2 hours) and retry when battery is >50%.

### Historical Analysis (Superseded)

The Windows Bluetooth LE stack aggressively caches GATT handle mappings after pairing. The sequence that triggers the bug:

1. First `dotnet test` process: connects, subscribes CCCD → **works** (fresh connection, no cache)
2. Process exits, Windows keeps the BLE connection alive in the background (paired device)
3. Second `dotnet test` process: connects, calls `GetGattServicesForUuidAsync(Uncached)` → gets characteristics
4. Writes CCCD again → returns `Success` (but Windows may use cached subscription that the device already invalidated)
5. The glasses, having been power-cycled, no longer have the CCCD subscription active on their end
6. Windows thinks notifications are enabled (from cache); glasses think they're not → **silent data loss**

### Contributing Factor: Glasses Per-Connection CCCD State

BLE specification says CCCD subscriptions are per-connection. When the glasses power cycle, they lose all subscription state. But Windows may reconnect using a bonded link key, making it appear to be the "same" connection from Windows' perspective while the glasses treat it as new.

### Alternative Theory: Notification Queue Exhaustion

Some BLE peripherals limit the number of pending notifications. If the glasses enter a state where their notification queue is full (e.g., from previous EnterTransferMode producing data), they may silently drop subsequent notifications until the queue is drained or the connection is fully torn down.

## Reproduction Steps

1. Run any test that successfully receives a notification (`DiagBleSetDeviceMode_CaptureSSIDFromNotify`)
2. Immediately run another test using the same fixture → 50%+ chance of 0 notifications
3. Power cycle glasses
4. Run test again → Still 0 notifications (Windows cache issue)

## Proposed Fixes

### Fix A: Force GATT Cache Clear on Reconnect (Recommended)

Before subscribing to notifications, explicitly clear and re-discover:

```csharp
// After getting _rxCharacteristic, force-read CCCD to verify actual state
var cccdResult = await _rxCharacteristic
    .ReadClientCharacteristicConfigurationDescriptorAsync()
    .AsTask(ct);

if (cccdResult.Status == GattCommunicationStatus.Success 
    && cccdResult.ClientCharacteristicConfigurationDescriptor != GattClientCharacteristicConfigurationDescriptorValue.Notify)
{
    _log.LogWarning("CCCD was not set to Notify — re-writing...");
}

// Write CCCD (even if "already set" — force the ATT write to the device)
var notifyResult = await _rxCharacteristic
    .WriteClientCharacteristicConfigurationDescriptorAsync(
        GattClientCharacteristicConfigurationDescriptorValue.Notify)
    .AsTask(ct);
```

### Fix B: Unsubscribe + Resubscribe with Verification

Add a verification step that sends a known-good command and confirms a notification arrives:

```csharp
internal async Task<bool> VerifyNotificationsAsync(CancellationToken ct)
{
    var tcs = new TaskCompletionSource<bool>();
    void handler(GattCharacteristic s, GattValueChangedEventArgs e) => tcs.TrySetResult(true);
    
    _rxCharacteristic!.ValueChanged += handler;
    try
    {
        // Send GetBattery (0x02, 0x04) — always produces a response
        await SendCommandAsync(HeyCyanCommands.GetBattery(), ct);
        var completed = await Task.WhenAny(tcs.Task, Task.Delay(3000, ct));
        return completed == tcs.Task;
    }
    finally
    {
        _rxCharacteristic.ValueChanged -= handler;
    }
}
```

If verification fails, tear down and reconnect:

```csharp
if (!await VerifyNotificationsAsync(ct))
{
    _log.LogWarning("Notification verification failed — forcing full reconnect");
    await DisconnectAsync();
    await Task.Delay(2000, ct); // Let Windows release the BLE handle
    await ConnectAsync(device, ct); // Fresh connection
}
```

### Fix C: Remove Device Pairing (Nuclear Option)

If the BLE cache is truly corrupt, programmatically unpair before connecting:

```csharp
var pairing = _bleDevice.DeviceInformation.Pairing;
if (pairing.IsPaired)
{
    await pairing.UnpairAsync().AsTask(ct);
    await Task.Delay(3000, ct); // Let Windows clean up
    // Re-discover device and connect fresh
}
```

### Fix D: Bypass Notifications Entirely (WiFi Transfer Path)

For the WiFi transfer feature specifically, we don't need notifications at all:
1. Send `EnterTransferMode` via BLE write (works reliably)
2. Scan WiFi for the glasses hotspot (no BLE notifications needed)
3. Connect with known password `123456789`
4. HTTP transfer files

This is the approach documented in RCA-806 Phase A. The WiFi scan approach doesn't depend on BLE notifications.

## Impact Assessment

| Feature | Blocked? | Workaround |
|---------|----------|------------|
| WiFi Transfer (get SSID via BLE) | Yes | Use WiFi scan instead (RCA-806 Phase A) |
| Battery level query | Partially | Returns 0% on timeout (graceful degradation) |
| Mode change confirmation | Partially | Fire-and-forget with delay works |
| Photo/video capture commands | No | Commands work, just no ACK confirmation |
| Audio streaming | No | Uses separate Classic BT A2DP/HFP path |

## Immediate Next Steps

1. **Accept the WiFi scan approach** — Don't depend on BLE notifications for WiFi transfer
2. **Investigate why Phase A (WiFi scan) didn't find a hotspot** — This is now the critical blocker (separate from notification issue)
3. **Add notification verification** to `ConnectAsync` with automatic retry (Fix B above)
4. **Log CCCD read-back** after subscription to detect silent failures
