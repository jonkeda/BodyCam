# RCA-805: QCSDK Binary Protocol Analysis — `openWifiWithMode:` Discovery

## Status: BREAKTHROUGH — Missing Command Identified

## Executive Summary

Reverse-engineering the iOS QCSDK binary reveals that the HeyCyan glasses support
**two separate BLE commands** for transfer mode:

1. `setDeviceMode:Transfer` — Sets the glasses into transfer mode (what we currently send)
2. **`openWifiWithMode:Transfer`** — Activates the WiFi hotspot and returns **SSID + password** via BLE

Our Windows code only sends command #1. The iOS SDK calls command #2 to receive WiFi credentials,
then connects to the glasses hotspot directly — **no WiFi Direct needed**.

## Key Findings

### 1. iOS SDK Method Surface (QCSDKCmdCreator)

```objc
// What we currently do (Android CyanBridge approach):
+ (void)setDeviceMode:(QCOperatorDeviceMode)mode
               success:(void (^)(void))suc
                  fail:(void (^)(NSInteger))fail;

// What we're MISSING (iOS approach — returns SSID/password!):
+ (void)openWifiWithMode:(QCOperatorDeviceMode)mode
                  success:(void (^)(NSString *ssid, NSString *password))suc
                     fail:(void (^)(NSInteger))fail;

// Also available — gets IP after WiFi is up:
+ (void)getDeviceWifiIPSuccess:(nullable void (^)(NSString *_Nullable))suc
                        failed:(nullable void (^)(void))fail;
```

### 2. iOS WiFi Transfer Sequence (from WiFiTransferManager.m)

```
Step 1: [QCSDKCmdCreator openWifiWithMode:QCOperatorDeviceModeTransfer ...]
        → Success callback receives: (NSString *ssid, NSString *password)
Step 2: Wait for hotspot readiness via getDeviceWifiIPSuccess: (retry 10x, 2s apart)
Step 3: getDeviceConfigWithFinished: (verify hotspot mode)
Step 4: NEHotspotConfiguration with received SSID + password
Step 5: Wait 5s for iOS to join hotspot
Step 6: HTTP test to glasses IP
Step 7: Begin media transfer
```

**Critical**: The iOS SDK does NOT use WiFi Direct/P2P at all!
It uses a standard WiFi hotspot that the phone joins.

### 5. QGSetGlassesModelCmd Internal Structure

**Instance variables** (all found in binary):
- `_cmdType` — differentiates setDeviceMode vs openWifi vs other ops
- `_currentStage` — tracks multi-step command progress
- `_mode` — QCOperatorDeviceMode value
- `_wifiName` — **WiFi SSID from glasses response**
- `_wifiPassword` — **WiFi password from glasses response**
- `_ipStr` — **IP address string**
- `OpenWifiSuccess` — success callback block
- `DevieModeFail`, `DevieModeSuccess` — mode callbacks
- `_resendCount`, `_modelFailResult` — retry handling

**Methods**:
- `operateDeviceType:mode:success:fail:` — base method (sends command)
- `sendCmdOnStage:` — constructs and sends BLE packet per stage
- `receiveData:Peripheral:Characteristic:` — parses BLE notifications
- `failWithError:` / `timeOut` — error handling

**sendCmdOnStage: local variables** (response parsing):
- `nameLen`, `nameData` — WiFi SSID length + data
- `psLen`, `psData` — WiFi password length + data
- `ip0`, `ip1`, `ip2`, `ip3` — IP address bytes
- `cmd`, `dataLen`, `sendDataCmd` — command construction

### 6. BLE Services Confirmed

The iOS SDK uses the SAME BLE service as our Windows code:
- Service: `de5bf728-d711-4e47-af26-65e3012a5dc7`
- Write: `de5bf72a-d711-4e47-af26-65e3012a5dc7`
- Notify: `de5bf729-d711-4e47-af26-65e3012a5dc7`

Also referenced: `6e40fff0` (Nordic UART Service) — possibly used by OdmCmd protocol.

**NOT referenced**: ae01/ae02/ae03/ae30 — iOS SDK does not use these services.

## Unknown: Exact Command Bytes for openWifiWithMode

The exact BLE bytes that `openWifiWithMode:` sends remain unknown. Two protocol layers exist:

**Protocol A (CyanBridge/Android)**: Raw serial commands
- `{0x02, 0x01, 0x04}` = setDeviceMode:Transfer
- This starts WiFi Direct/P2P on glasses

**Protocol B (QCSDK/iOS)**: OdmCmd protocol with ODM_DFU_Operation opcodes
- `0x41` = SetDeviceMode opcode
- Possibly packaged via `QCDFU_Utils.packageData:type:` with length framing
- The `operateDeviceType:mode:` base method sends different commands for setDeviceMode vs openWifi

### Hypotheses for openWifiWithMode command bytes

**H1**: Same command `{0x02, 0x01, 0x04}` but multi-stage — WiFi info arrives in a LATER notification  
**H2**: Different deviceType byte: `{0x02, 0x02, 0x04}` or `{0x02, 0x03, 0x04}`  
**H3**: OdmCmd format: `{0x41, 0x04}` (tried before, no response — may need framing)  
**H4**: OdmCmd with framing: `{length_hi, length_lo, 0x41, 0x04}` or `{0x41, 0x00, 0x01, 0x04}`  
**H5**: Command sent to NUS service (`6e40fff0`) instead of serial port  

### WiFi Info Response Format (inferred from local variables)

```
[header 7B][...][nameLen 1B][nameData nameLen B][psLen 1B][psData psLen B]
```
Or possibly with IP:
```
[header 7B][ip0][ip1][ip2][ip3][nameLen 1B][nameData...][psLen 1B][psData...]
```

## Test Plan (requires charged glasses)

### Test 1: Extended Notification Capture
Send `{0x02, 0x01, 0x04}`, listen for ALL notifications for 90 seconds.
Parse any notification >11 bytes for nameLen/nameData/psLen/psData.

### Test 2: Alternative deviceType Values
Try `{0x02, N, 0x04}` for N in {0x02, 0x03, 0x05, 0x07, 0x08, 0x0A}.
Listen for responses on notify characteristic.

### Test 3: OdmCmd Opcode Variations
Try raw opcode: `{0x41, 0x04}`, `{0x41, 0x01, 0x04}`
Try with length prefix: `{0x00, 0x01, 0x41, 0x04}`, `{0x00, 0x02, 0x41, 0x01, 0x04}`

### Test 4: NUS Service Write
Send commands via `6e400002` (NUS write) instead of `de5bf72a`.

### Test 5: Hotspot Brute Force
After sending `{0x02, 0x01, 0x04}`, scan for WiFi networks matching:
- `DIRECT-*M01*`
- `M01*`
- `HeyCyan*`
Try connecting with password `123456789`.

### Test 6: Full Notification Hex Dump
Capture and hex-dump EVERY notification received after any command,
especially looking for ASCII strings that could be SSIDs.

## Impact

If we can identify the correct command for `openWifiWithMode:`, we can:
1. Get SSID + password directly from glasses via BLE
2. Connect to glasses hotspot via regular WiFi (no WiFi Direct needed!)
3. Completely bypass the broken Windows WiFi Direct stack
4. Use standard `HttpClient` for media transfer

This would resolve the entire Phase 8 WiFi transfer blocker.
